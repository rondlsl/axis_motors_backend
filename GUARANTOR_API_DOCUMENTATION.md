# Документация API функционала "Гарант"

## Обзор
Функционал "Гарант" позволяет пользователям, которым было отказано в регистрации, воспользоваться услугой гаранта - человека, который в случае ДТП или других обязательств несёт материальную ответственность.

## Основные эндпоинты

### 1. Информация о функции гаранта
```
GET /guarantor/info
```
Возвращает информацию о том, что такое гарант (для кнопки "?" в интерфейсе).

**Ответ:**
```json
{
  "title": "Что такое Гарант?",
  "description": "Гарант — лицо, которое в случае ДТП несёт материальную ответственность",
  "details": [
    "Гарант - это человек, который берет на себя материальную ответственность за ваши действия",
    "В случае ДТП или других обязательств, гарант будет нести финансовую ответственность",
    "Для оформления услуги гарант должен подписать специальный договор",
    "После подписания договора гаранта, также подписывается договор субаренды автомобиля"
  ]
}
```

### 2. Создание заявки на гаранта
```
POST /guarantor/request
Authorization: Bearer <token>
```

**Тело запроса:**
```json
{
  "guarantor_info": {
    "full_name": "Иванов Иван Иванович",
    "phone_number": "77001234567"
  },
  "reason": "Причина отказа в регистрации (необязательно)"
}
```

**Ответы:**
- Если пользователя нет в системе:
```json
{
  "message": "Пользователь не найден. SMS приглашение отправлено.",
  "user_exists": false,
  "sms_result": {...}
}
```

- Если пользователь найден:
```json
{
  "message": "Заявка на гаранта создана успешно",
  "user_exists": true,
  "request_id": 123,
  "guarantor_name": "Иванов Иван Иванович"
}
```

### 3. Ответ на заявку гаранта
```
POST /guarantor/respond/{request_id}
Authorization: Bearer <token>
```

**Тело запроса:**
```json
{
  "accept": true,
  "rejection_reason": "Причина отказа (если accept: false)"
}
```

### 4. Получение информации о гарантах пользователя
```
GET /guarantor/my-info
Authorization: Bearer <token>
```

**Ответ:**
```json
{
  "sent_requests": [...],        // Заявки, которые я отправил
  "received_requests": [...],    // Заявки, которые я получил  
  "my_clients": [...],          // Люди, за которых я ручаюсь
  "my_guarantors": [...]        // Мои гаранты
}
```

### 5. Подписание договора
```
POST /guarantor/sign-contract
Authorization: Bearer <token>
```

**Тело запроса:**
```json
{
  "contract_type": "guarantor",  // "guarantor" или "sublease"
  "guarantor_relationship_id": 123
}
```

### 6. Проверка платежеспособности пользователя
```
POST /guarantor/check-user-eligibility
Authorization: Bearer <token>
```

**Тело запроса:**
```json
{
  "phone_number": "77001234567"
}
```

**Ответ:**
```json
{
  "user_exists": true,
  "user_id": 123,
  "is_eligible": true,
  "has_car_access": true,
  "user_name": "Иванов Иван Иванович",
  "reason": null
}
```

### 7. Получение списка договоров
```
GET /guarantor/contracts
Authorization: Bearer <token>
```

**Ответ:**
```json
{
  "guarantor_contracts": [...],  // Договоры гаранта
  "sublease_contracts": [...]    // Договоры субаренды
}
```

## Административные эндпоинты

### 1. Загрузка договоров (только администраторы)
```
POST /guarantor/upload-contract/{contract_type}
Authorization: Bearer <admin_token>
Content-Type: multipart/form-data
```

**Параметры:**
- `contract_type`: "guarantor" или "sublease"
- `file`: PDF или DOC/DOCX файл договора

### 2. Отклонение пользователя с предложением гаранта
```
POST /admin/approve-user
Authorization: Bearer <admin_token>
```

**Тело запроса:**
```json
{
  "user_id": 123,
  "approved": false,
  "rejection_reason": "Причина отказа"
}
```

### 3. Получение пользователей, ожидающих одобрения
```
GET /admin/pending-users
Authorization: Bearer <admin_token>
```

## Логика работы

### 1. При регистрации пользователя
- Если пользователю отказано, администратор может отправить SMS с предложением гаранта
- SMS содержит: "Вам отказано. Причина: [текст причины]. Однако вы можете воспользоваться услугой «Гарант»..."

### 2. Добавление гаранта
- Пользователь заполняет данные гаранта (ФИО, телефон)
- Если гарант не зарегистрирован - отправляется SMS приглашение
- Если гарант зарегистрирован - создается заявка и проверяется его платежеспособность

### 3. Принятие заявки гарантом
- Гарант получает уведомление о заявке
- При принятии заявки создается активная связь гарант-клиент
- Требуется подписание договора гаранта, затем договора субаренды

### 4. SMS уведомления
- При указании незарегистрированного гаранта: "[ФИО] выбрал(а) вас в качестве Гаранта. Перейдите по ссылке и скачайте приложение"
- При отказе в регистрации: информация об услуге гаранта

## Тестирование

Для тестирования SMS:
- Установите `SMS_TOKEN=6666` в переменных окружения
- При этом значении SMS будут выводиться в консоль вместо отправки

## Файлы и директории

Созданные файлы:
- `app/models/guarantor_model.py` - модели БД
- `app/guarantor/schemas.py` - Pydantic схемы
- `app/guarantor/router.py` - основной роутер API
- `app/guarantor/sms_utils.py` - утилиты для SMS
- `app/admin/router.py` - административные функции

Директории для файлов:
- `uploads/contracts/guarantor/` - договоры гаранта
- `uploads/contracts/sublease/` - договоры субаренды

## Swagger документация

После запуска приложения, Swagger документация будет доступна по адресу:
```
http://localhost:8000/docs
```

Все эндпоинты функционала гарантов будут в разделах:
- **Guarantor** - основные функции
- **Admin** - административные функции
