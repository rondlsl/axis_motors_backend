# Анализ Sequence: Поддержка через Telegram

Детальный анализ sequence-диаграммы взаимодействия пользователя с поддержкой через Telegram-бота и веб-панель оператора.

---

## 2.1. Общее описание процесса

Поддержка реализована как двусторонний канал: **клиент** общается с ботом в Telegram (личный чат), **оператор** работает в веб-панели и/или в Telegram-группе. Сообщения клиента сохраняются в БД (SupportChat, SupportMessage), оператор видит обращения через REST API и веб-интерфейс; ответы оператора из веб-панели отправляются клиенту в Telegram. Дополнительно операторы могут отвечать прямо из Telegram-группы по кнопке «Ответить». Неактивные чаты в статусе «Решено» автоматически закрываются через 12 часов.

На диаграмме показаны три основных потока: (1) входящее сообщение от пользователя → сохранение в БД и уведомление операторам, (2) ответ оператора через API → сохранение и доставка в Telegram клиенту, (3) фоновая задача авто-закрытия чатов.

---

## 2.2. Участники

| Участник | Роль |
|----------|------|
| **Telegram User (TU)** | Клиент: пишет боту в личном чате Telegram. |
| **Telegram API (TG)** | Доставка сообщений между клиентом и ботом; в коде используется **long polling** (не webhook). |
| **Support Bot (BOT)** | Обработка входящих сообщений (python-telegram-bot), создание/поиск чата, запись сообщений в БД, уведомление группы поддержки в Telegram. |
| **PostgreSQL (DB)** | Хранение SupportChat, SupportMessage; связь с User по номеру телефона (azv_user_id). |
| **WebSocket (WS)** | На диаграмме — доставка уведомления оператору о новом сообщении. В текущей реализации уведомления операторам идут **в Telegram-группу** (SUPPORT_GROUP_ID), отдельный WebSocket для «новое сообщение в чате» не реализован. |
| **Оператор (OP)** | Сотрудник поддержки: веб-панель (REST API) и/или Telegram-группа (кнопка «Ответить»). |

**Примечание:** Участник «API» на диаграмме явно не назван; по смыслу это FastAPI-приложение (тот же процесс, что и BOT, но REST-эндпоинты модуля поддержки).

## 2.3. Пошаговый разбор

### 1. Входящее сообщение от пользователя

| Шаг | Описание | Соответствие коду |
|-----|----------|-------------------|
| TU → TG | Пользователь отправляет сообщение боту в Telegram. | Реализовано (личный чат с ботом). |
| TG → BOT | **На диаграмме:** Webhook → `/support/webhook`. | **Расхождение:** в коде webhook не используется; бот получает обновления через **long polling** (`application.updater.start_polling()` в `telegram_bot.py`). Эндпоинта `/support/webhook` нет. |
| BOT → DB | Найти или создать SupportChat; создать SupportMessage (sender_type=client). | Частично: для **уже существующего** чата (пользователь в состоянии IN_CHAT) — `get_chat_by_telegram_id`, затем `support_service.add_message`. Для **нового** обращения чат не создаётся «по одному сообщению»: сначала пошаговый сбор ФИО, телефона и текста (S3 в business_logic_algorithms.md), затем `create_chat` и первое сообщение. |
| BOT → WS | Notify: новое сообщение. | **Расхождение:** в коде уведомление операторам идёт в **Telegram-группу** (SUPPORT_GROUP_ID) через `send_message_notification_to_support_group` и т.п., а не через WebSocket. WebSocket-эндпоинты `/ws/support/*` используются для списков (машины, пользователи), а не для событий «новое сообщение в чате». |
| WS → OP | WebSocket push оператору. | В реализации оператор узнаёт о новом сообщении из сообщения в Telegram-группе (с кнопкой «Ответить») или при обновлении списка чатов в веб-панели. |

### 2. Ответ оператора

| Шаг | Описание | Соответствие коду |
|-----|----------|-------------------|
| OP → API | POST `/support/chats/{id}/messages`. | **Расхождение:** в коде эндпоинт — **POST /api/support/messages** с телом `{ "chat_id": "<sid>", "message_text": "..." }`, а не путь вида `/chats/{id}/messages`. |
| API → DB | Создать SupportMessage (sender_type=support). | Реализовано: `support_service.add_message(message_create, sender_user_id=current_user.id)`. Обновляется также `chat.updated_at`. |
| API → TG | Отправить ответ пользователю в Telegram. | Реализовано: `send_message_to_client(chat.user_telegram_id, message_data.message_text)` — POST sendMessage с префиксом «Поддержка:\n\n». Длинные сообщения (>4096) разбиваются на части. |
| TG → TU | Ответ оператора доставлен клиенту. | Реализовано. |

### 3. Автоматическое закрытие чата (примечание на диаграмме)

| Описание на диаграмме | Реализация в коде |
|----------------------|-------------------|
| «Через 12 часов без активности» → auto_close_support_chats → closed. | Задача по расписанию **раз в час**. Закрываются чаты в статусе **RESOLVED**, у которых `updated_at` старше **12 часов** (т.е. 12 часов с последнего обновления при уже решённом обращении). Устанавливаются `status=CLOSED`, `closed_at=now`. Клиенту отправляется уведомление в Telegram об авто-закрытии. |

## 2.4. Анализ потоков данных и состояний

### Поля SupportChat и их изменение

| Поле | Когда меняется |
|------|-----------------|
| **status** | NEW (при создании) → IN_PROGRESS (при назначении оператора или первом ответе из группы) → RESOLVED (ручное) → CLOSED (ручное или авто-закрытие через 12 ч для RESOLVED). |
| **assigned_to** | При «взять в работу» (take) или назначении другим оператором (assign) — только для чатов в статусе NEW. |
| **updated_at** | При создании чата, при добавлении сообщения (`add_message`), при смене статуса/назначении, при авто-закрытии. |
| **closed_at** | Устанавливается при переходе в CLOSED (ручное или авто-закрытие). |

Дополнительно в модели: user_telegram_id, user_telegram_username, user_name, user_phone, azv_user_id, created_at.

### Идентификация пользователя в Telegram

- **user_telegram_id** (SupportChat) — числовой ID пользователя в Telegram (`update.effective_user.id`). Связь «один активный чат на telegram_id» для статусов NEW/IN_PROGRESS/RESOLVED.
- **azv_user_id** — связь с таблицей User: при создании чата ищется пользователь по `User.phone_number` (нормализованный номер без «+»); при совпадении с введённым клиентом телефоном в SupportChat записывается `azv_user_id`.
- В SupportMessage сохраняются **telegram_message_id** и **telegram_chat_id** (чат Telegram, для лички совпадает с идентификатором диалога с пользователем).

Итого: пользователь в Telegram идентифицируется по **user_telegram_id**; привязка к учётной записи AZV — по номеру телефона через **azv_user_id**.

### Формат WebSocket-уведомления

На диаграмме предполагается, что WS доставляет оператору событие о новом сообщении. В текущей кодовой базе **такого WebSocket-события для чатов поддержки нет**. Существующие каналы:

- **Telegram-группа:** текст/медиа с данными чата и кнопкой «Ответить» (callback_data `reply_{chat.sid}`).
- **REST:** оператор открывает список чатов или конкретный чат (GET /api/support/chats, GET /api/support/chats/{id}, GET /api/support/chats/{id}/messages).

Если в ТЗ закладывается real-time уведомление в веб-панели, формат можно описать, например: `{ "type": "support_new_message", "chat_id": "<sid>", "preview": "..." }` по существующей схеме WebSocket (подписка по роли/пользователю).

## 2.5. Критическая оценка

### Полнота диаграммы

- **Отражено:** входящее сообщение → БД → уведомление; ответ оператора через API → БД → Telegram; авто-закрытие через 12 ч.
- **Не отражено:**
  - **Создание обращения:** многошаговый сценарий (ФИО, телефон, текст), затем создание чата и первого сообщения; кнопка «Обратиться в техподдержку» и состояния (WAITING_FOR_NAME, WAITING_FOR_PHONE, WAITING_FOR_MESSAGE, IN_CHAT).
  - **Медиа:** загрузка фото/документов/голосовых, сохранение в MinIO, запись в SupportMessage (media_type, media_url и т.д.), отправка в группу поддержки.
  - **Назначение оператора:** взять чат (take), назначить другому (assign), смена статуса (NEW → IN_PROGRESS, RESOLVED, CLOSED).
  - **Ответ из группы Telegram:** кнопка «Ответить» в группе, режим support_reply_states, отправка следующего текста клиенту и запись SupportMessage(sender_type=SUPPORT).
  - **Оценка качества:** в коде не описана отдельным сценарием; на диаграмме нет.
  - **Ручное закрытие/смена статуса** и уведомление клиенту (RESOLVED/CLOSED).

### Безопасность

- **Вебхук Telegram:** на диаграмме указан webhook; в коде его нет. При включении webhook в ТЗ следует описать проверку секретного токена (secret_token) в заголовке запроса от Telegram.
- **API эндпоинты:** все маршруты `/api/support/*` защищены: `require_support_role` или `require_admin_role` (JWT). Для отправки сообщения дополнительно проверяется, что чат назначен текущему оператору (assigned_to == current_user.id для роли SUPPORT).

### Масштабирование

- **Один бот на всех пользователей:** один процесс бота с long polling обрабатывает всех клиентов; состояние (user_states, support_reply_states) хранится в памяти. При нескольких инстансах приложения нужна синхронизация состояния или отказ от in-memory состояний (например, хранение состояния в БД/Redis). Ограничение Telegram: один бот — один webhook/polling endpoint.
- **Группа поддержки:** все операторы получают уведомления в одну Telegram-группу; разграничение «кто что видит» — через веб-панель (фильтр по assigned_to).

### Соответствие коду и алгоритмам домена «Поддержка»

- **S1 (polling):** бот запускается через polling, не webhook — диаграмму стоит скорректировать.
- **S3 (создание обращения):** пошаговый сбор данных и создание чата — на диаграмме не отражён.
- **S4, S5 (сообщения и медиа):** добавление сообщения и обновление chat.updated_at — отражено упрощённо; медиа и MinIO — нет.
- **S6 (уведомления в группу):** реализовано через Telegram-группу, а не WebSocket — диаграмму уточнить.
- **S7 (ответ из группы):** ответ оператора из Telegram-группы — на диаграмме нет.
- **S11 (REST: ответ оператора):** путь и метод в коде — POST /api/support/messages с chat_id в теле — на диаграмме указан другой путь.
- **S12 (авто-закрытие):** только чаты в статусе RESOLVED и updated_at старше 12 часов — на диаграмме лучше явно указать «RESOLVED» и «12 ч с последнего обновления».

## 2.6. Рекомендации

### Что добавить в диаграмму для полноты ТЗ

1. **Вход в систему поддержки:** заменить «Webhook → /support/webhook» на **Long Polling** (Telegram → BOT) и при необходимости указать, что при переходе на webhook будет использоваться проверка secret_token.
2. **Создание обращения:** отдельный фрагмент: сбор ФИО, телефона, текста (сообщения в личку боту) → создание SupportChat и первого SupportMessage → уведомление в группу с кнопкой «Ответить».
3. **Уведомление операторам:** явно указать **Telegram-группу** (SUPPORT_GROUP_ID); при наличии или планировании WebSocket для веб-панели — добавить участника и сообщение «Notify (new message)» с пометкой «опционально / в разработке».
4. **Ответ оператора:** указать реальный эндпоинт **POST /api/support/messages** с телом `{ chat_id, message_text }`.
5. **Альтернативный канал ответа:** ответ из Telegram-группы (кнопка «Ответить» → следующее сообщение оператора уходит клиенту и пишется в БД).
6. **Медиа:** опциональный поток: клиент отправляет фото/файл → бот сохраняет в MinIO и SupportMessage (media_*) → уведомление в группу (с медиа и кнопкой).
7. **Авто-закрытие:** уточнить условие: «Чаты в статусе RESOLVED, у которых updated_at старше 12 часов → status=CLOSED, closed_at; уведомление клиенту в Telegram».
8. **Назначение и статусы:** при необходимости ТЗ — отдельные шаги: take/assign (assigned_to, IN_PROGRESS), смена статуса (RESOLVED/CLOSED), уведомление клиенту при RESOLVED/CLOSED.

### Детали реализации для уточнения в ТЗ

- **Webhook vs Polling:** зафиксировать, остаётся ли polling или планируется webhook (и URL, проверка secret_token).
- **Real-time в веб-панели:** нужен ли WebSocket (или аналог) для событий «новое сообщение» / «новый чат» для операторов; при положительном решении — формат сообщений и подписка.
- **Масштабирование бота:** один инстанс или несколько; при нескольких — где хранить user_states/support_reply_states (Redis/БД).
- **Оценка качества обращения:** требуется ли и где хранить (отдельная модель, поле в SupportChat).
- **Идемпотентность:** при повторной отправке одного и того же сообщения из веб-панели (сетевой повтор) — нужна ли защита (например, idempotency key для POST /api/support/messages).

После этих уточнений диаграмма будет соответствовать реализации и может использоваться в ТЗ как описание потока «Поддержка через Telegram».
