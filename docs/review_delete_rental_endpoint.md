# Code Review: DELETE /admin/rentals/{rental_id}

## 1. Общая оценка endpoint'а

Endpoint выполняет hard-delete аренды и связанных сущностей в одной транзакции. Логика удаления по порядку (дети → родитель) корректна. Есть риски по race conditions, утечке внутренних ошибок и отсутствию аудита.

---

## 2. Что сделано правильно

- **Порядок удаления** — сначала дочерние сущности (WalletTransaction, UserContractSignature, RentalAction, RentalReview), затем RentalHistory; FK не нарушаются.
- **Одна транзакция** — все `db.delete()` и один `db.commit()`; при любом исключении до commit вызывается `db.rollback()` → **атомарность в рамках одной транзакции есть**.
- **Права доступа** — проверка `ADMIN` / `SUPPORT`.
- **Валидация ID** — `safe_sid_to_uuid`, 400 при неверном формате, 404 при отсутствии аренды.
- **HTTPException** — перебрасывается в `except HTTPException: raise`, не затирается.

---

## 3. Проблемы и риски

### 3.1 Атомарность и транзакции

| Вопрос | Ответ |
|--------|--------|
| Гарантируется ли полная атомарность? | **Да** — один commit в конце; при любой ошибке до commit — rollback. Partial delete в БД не остаётся. |
| Ошибка после удаления 2–3 сущностей? | Все delete только помечают объекты в сессии; физическое удаление в БД происходит при commit. До commit — откат. |
| Нужен ли `with db.begin()`? | Нет. Текущая схема (commit в конце + rollback в except) достаточна. |
| Возможен ли partial delete? | **Нет** — либо все изменения коммитятся, либо откатываются. |

**Severity:** нет критичных проблем.

---

### 3.2 Race conditions

| Сценарий | Риск |
|----------|------|
| **Два админа удаляют одну аренду** | Оба читают rental без блокировки. Первый commit удаляет всё. Второй commit пытается удалить уже удалённые строки — в PostgreSQL DELETE по несуществующему id не даёт ошибку. Второй запрос вернёт 200 и «успех». Идемпотентность по факту есть. |
| **Billing job параллельно** | Billing читает RentalHistory, потом создаёт WalletTransaction(related_rental_id=…). Между чтением и записью админ может удалить аренду и транзакции. Billing при insert получит **IntegrityError** (FK violation). Риск: падение billing job, без блокировки аренды при удалении — **реальный**. |
| **Повторный DELETE** | После первого успешного DELETE второй запрос получит 404 (аренда уже не найдена). Идемпотентность по коду есть. |

**Рекомендация:** блокировать строку аренды при удалении (`SELECT ... FOR UPDATE`), чтобы billing (и другие джобы) не могли параллельно читать/писать по этой аренде до завершения транзакции удаления.

**Severity:** **MEDIUM** (возможен IntegrityError в billing при высокой конкуренции).

---

### 3.3 Целостность данных

| Вопрос | Ответ |
|--------|--------|
| Все ли связанные сущности удаляются? | **Да** — учтены: WalletTransaction, UserContractSignature, RentalAction, RentalReview, RentalHistory. Других FK на `rental_history.id` в проекте нет. |
| FK constraints / orphan records? | Порядок удаления корректен, orphan’ов не остаётся. |
| ON DELETE CASCADE? | В моделях есть cascade только на actions и contract_signatures. Ручное удаление нужно для WalletTransaction и RentalReview; текущая реализация это делает. |
| Логи / аудит / история? | **Нет** — факт удаления ни в БД, ни в структурированный audit log не пишется. Только ответ клиенту. |

**Критичные данные:** удаляются безвозвратно: сама аренда, транзакции кошелька по ней, подписи договоров, действия, отзыв. Восстановление только из бэка БД.

**Severity:** **MEDIUM** (отсутствие аудита для финансово/юридически значимых операций).

---

### 3.4 Безопасность и права

| Вопрос | Ответ |
|--------|--------|
| Достаточно ли ADMIN/SUPPORT? | Да для внутреннего админского API. |
| Нужен ли audit (кто, когда удалил)? | **Желательно** — хотя бы логирование (admin_id, rental_id, timestamp). |
| deleted_by / reason / comment? | Для compliance может понадобиться; для минимального прод-релиза можно ограничиться логом. |
| Злоупотребление? | Низкий при контроле доступа; риск выше при утечке токена админа (как и у любого админского эндпоинта). |

**Severity:** **LOW** (улучшение — audit log / лог).

---

### 3.5 Обработка ошибок

| Вопрос | Ответ |
|--------|--------|
| Rollback корректен? | Да — `except Exception: db.rollback(); raise HTTPException(500, ...)`. |
| HTTPException не глотается? | Нет — `except HTTPException: raise`. |
| Информативность сообщения? | В 500 отдаётся `detail=f"Ошибка удаления аренды: {str(e)}"` — **утечка внутренних деталей** (сообщения БД, возможные фрагменты запросов). В проде лучше общее сообщение, а полный `str(e)` — только в лог. |
| Утечка internal details? | **Да** — `str(e)` в ответе клиенту. |

**Severity:** **MEDIUM** (информация об ошибках не должна уходить клиенту в проде).

---

## 4. Минимальные правки перед продом

1. **Не отдавать клиенту `str(e)`** — общий текст в `detail`, полный exception — в лог.
2. **Блокировать аренду при удалении** — `db.query(RentalHistory).filter(...).with_for_update().first()` чтобы снизить race с billing.
3. **Логировать факт удаления** — admin_id, rental_id (и при необходимости timestamp) в application log.

---

## 5. Итоговый вердикт

**NEEDS FIX** — перед продом обязательно:

1. Убрать утечку внутренней ошибки в 500 ответе.
2. Добавить блокировку строки аренды (FOR UPDATE) при удалении.
3. Добавить лог удаления (кто, какую аренду удалил).

**Документировать для команды:**

- Endpoint выполняет **hard-delete**; восстановление только из бэка.
- Аудит удалений реализован только через application-логи (и при необходимости нужно расширить до отдельной audit-таблицы).
- При высокой нагрузке и активном billing возможны редкие IntegrityError в billing при одновременном удалении аренды; FOR UPDATE уменьшает этот риск.
