# Похожие кейсы: баланс и транзакции во время поездки

## Итог проверки

**Единственный проблемный кейс** — верификация баланса при завершении аренды в `verify_and_fix_rental_balance` (уже исправлен в `app/rent/utils/balance_utils.py`). Других мест, где баланс перезаписывается на «ожидаемый» расчёт по подмножеству транзакций (игнорируя пополнения/бонусы во время поездки), не найдено.

---

## 1. Верификация при завершении аренды ✅ исправлено

- **Где:** `app/rent/utils/balance_utils.py` → `verify_and_fix_rental_balance`
- **Вызовы:** 
  - `POST /rent/complete` (app/rent/router.py)
  - Админка: завершение аренды за клиента (admin_end_rental) и отправка ревью (submit_rental_review_impl) в app/admin/users/router.py
- **Суть бага:** расчёт `expected_balance_after` шёл только по транзакциям аренды и пост-арендным; пополнения (DEPOSIT) и прочие операции во время поездки не учитывались → баланс перезаписывался на заниженный.
- **Исправление:** учёт всех транзакций пользователя за период аренды в хронологическом порядке (в т.ч. пополнения, бонусы и т.д.), затем пост-арендные. Один фикс закрывает все три точки входа (complete, admin end, review).

---

## 2. Ручной пересчёт баланса (админ / саппорт) — ок

- **Где:**  
  - `POST /admin/users/{user_id}/balance/recalculate` (app/admin/users/router.py)  
  - аналог в app/support/users/router.py
- **Логика:** перебор **всех** транзакций пользователя по `created_at`, обновление `balance_before`/`balance_after` и `user.wallet_balance`.
- **Вывод:** пополнения и любые другие транзакции учитываются, отдельного кейса «пополнение во время поездки» нет.

---

## 3. Биллинг во время поездки (process_rentals_sync) — ок

- **Где:** app/rent/utils/billing.py
- **Логика:** списание с баланса делается как `current_balance = user.wallet_balance`, затем `user.wallet_balance = current_balance - charge` (или через `record_wallet_transaction` и обновление баланса). Нигде не пересчитывается «ожидаемый» баланс из транзакций.
- **Вывод:** используется актуальный баланс из БД (уже с пополнением). Похожего бага нет.

---

## 4. Проверки «достаточно средств» — ок

- **Где:** старт/продление аренды, отмена, доставка и т.д. (router, admin, support).
- **Логика:** везде проверка вида `current_user.wallet_balance < required_balance` или `target_user.wallet_balance < ...`, т.е. текущее значение из БД.
- **Вывод:** отдельного «прогноза» баланса без учёта пополнения нет; после пополнения баланс уже обновлён в БД.

---

## 5. Корректировки списаний в complete_rental и в review — ок

- **Где:** app/rent/router.py (complete_rental), app/admin/users/router.py (submit_rental_review_impl).
- **Логика:** перерасчёт сумм (минуты, база, овертайм и т.д.) с выражениями вида `balance_before_correction = current_user.wallet_balance + abs(old_charge)`, затем `user.wallet_balance = balance_before_correction - expected_cost`. Используется текущий баланс пользователя (уже с пополнением).
- **Вывод:** сами эти блоки пополнение не «съедают». Раньше проблему создавала только последующая верификация в `verify_and_fix_rental_balance`, которая перезаписывала баланс. После фикса в balance_utils цепочка корректна.

---

## 6. Другие операции с балансом

- Пополнение (DEPOSIT), бонусы (PROMO_BONUS), штрафы, возвраты и т.д. — везде либо инкремент (`wallet_balance += ...` / `-= ...`), либо явное присвоение на основе текущего баланса/суммы операции. Нет пересчёта «ожидаемого» баланса по подмножеству транзакций.

---

## Рекомендация

- Доверить исправлению в `balance_utils.py` все сценарии завершения поездки (клиент, админ, ревью).
- При добавлении нового кода, который «восстанавливает» или «верифицирует» баланс пользователя по транзакциям, строить цепочку по **всем** транзакциям пользователя в хронологическом порядке (или явно документировать, почему учитывается только подмножество).
