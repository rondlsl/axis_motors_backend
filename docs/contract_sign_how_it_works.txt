================================================================================
КАК РАБОТАЕТ ПОДПИСАНИЕ ДОГОВОРОВ (contract_sign)
================================================================================

0. ОТКУДА ДОСТАЮТСЯ И ГДЕ ХРАНЯТСЯ
--------------------------------------------------------------------------------

  • Типы договоров (ContractType)
    Откуда: enum в коде, app/models/contract_model.py (ContractType).
    Где хранится: нигде в БД отдельно не хранится — только как значение поля contract_type в contract_files и при запросах (фильтр по типу).

  • Шаблоны договоров (файлы PDF/DOC/DOCX)
    Откуда: загружает админ через POST /contracts/upload (файл + contract_type).
    Где хранится:
      - Файл: MinIO, бакет из конфига (MINIO_BUCKET_UPLOADS), папка "contracts", имя вида {contract_type.value}_{uuid}.pdf.
      - Метаданные: таблица contract_files (PostgreSQL).
        Поля: id, contract_type, file_path (URL в MinIO), file_name, is_active, uploaded_at, updated_at.
    При новой загрузке договора того же типа старые записи этого типа помечаются is_active=False; активен только один файл на тип.

  • Подписи пользователей
    Откуда: создаются при POST /contracts/sign (и админ/support/гарант эндпоинтах).
    Где хранится: таблица user_contract_signatures (PostgreSQL).
      Поля: id, user_id (FK users), contract_file_id (FK contract_files), rental_id (FK rental_history, nullable), guarantor_relationship_id (FK guarantors, nullable), digital_signature, signed_at.
    Цифровая подпись (digital_signature) при создании подписи берётся из users.digital_signature (не из тела запроса).

  • Цифровая подпись пользователя (для подписания)
    Откуда: генерируется/сохраняется при регистрации/обновлении профиля (поле User.digital_signature).
    Где хранится: таблица users, столбец digital_signature. В подпись договора подставляется текущее значение этого поля.

  • Связь с арендой и гарантом
    rental_id достаётся из тела запроса (SID аренды), проверяется по таблице rental_history.
    guarantor_relationship_id — из тела запроса (SID связи гарант–клиент), проверяется по таблице guarantors.

Итого:
  - Типы договоров: код (enum).
  - Файлы шаблонов: MinIO (папка contracts) + запись в contract_files.
  - Факт подписания: user_contract_signatures (PostgreSQL).
  - Подпись как строка: users.digital_signature (копируется в user_contract_signatures при подписании).


1. ТИПЫ ДОГОВОРОВ (ContractType)
--------------------------------------------------------------------------------
Модель: app/models/contract_model.py

  GUARANTOR_CONTRACT        — Договор гаранта
  GUARANTOR_MAIN_CONTRACT   — Основной договор гаранта
  USER_AGREEMENT            — Пользовательское соглашение
  CONSENT_TO_DATA_PROCESSING — Согласие на обработку персональных данных
  MAIN_CONTRACT             — Договор присоединения
  RENTAL_MAIN_CONTRACT      — Основной договор аренды (привязан к rental_id)
  APPENDIX_7_1              — Приложение 7.1 (привязан к rental_id)
  APPENDIX_7_2              — Приложение 7.2 (акт возврата, привязан к rental_id)

Шаблоны хранятся в ContractFile: один активный файл (is_active=True) на тип.
Подписи — в UserContractSignature (user_id, contract_file_id, rental_id, guarantor_relationship_id, digital_signature, signed_at).


2. ОСНОВНОЙ ЭНДПОИНТ: POST /contracts/sign
--------------------------------------------------------------------------------
Файл: app/contracts/router.py, sign_contract()

Тело запроса (SignContractRequest):
  contract_type: ContractType (обязательно)
  rental_id: str | null — для RENTAL_MAIN_CONTRACT, APPENDIX_7_1, APPENDIX_7_2
  guarantor_relationship_id: str | null — для GUARANTOR_CONTRACT, GUARANTOR_MAIN_CONTRACT

Шаги:

  1) Найти активный ContractFile по contract_type (is_active=True).
     Если нет — 404 "Активный файл договора типа ... не найден".

  2) Проверить digital_signature у current_user. Если нет — 400.

  3) Парсинг rental_id и guarantor_relationship_id (UUID). При неверном формате — 400 + уведомление в Telegram.

  4) Проверка дубликата подписи:
     Ищем UserContractSignature по:
       user_id == current_user.id
       contract_file_id == contract_file.id
       rental_id совпадает (либо оба NULL)
       guarantor_relationship_id совпадает (либо оба NULL)
     Если найдена — 400 "Этот договор уже подписан" + лог в Telegram.

  5) Для типов НЕ RENTAL_MAIN_CONTRACT / APPENDIX_7_1 / APPENDIX_7_2:
     Проверка "тот же тип договора уже подписан" (по типу ContractFile + user + rental_id + guarantor_relationship_id).
     Если уже есть подпись того же типа (другой contract_file) — 400 "Этот тип договора уже подписан".

  6) Для RENTAL_MAIN_CONTRACT, APPENDIX_7_1, APPENDIX_7_2:
     - rental_id обязателен. Иначе — 400.
     - RentalHistory по rental_uuid должна существовать.
     - Если текущий пользователь — MECHANIC: он может подписывать только если он mechanic_inspector или delivery_mechanic этой аренды. Иначе 403.
     - Если обычный пользователь: rental.user_id должен совпадать с current_user.id. Иначе 403.

  7) Для GUARANTOR_CONTRACT, GUARANTOR_MAIN_CONTRACT:
     - guarantor_relationship_id обязателен.
     - Guarantor по id должна существовать и guarantor.guarantor_id == current_user.id. Иначе 403.

  8) Создание подписи:
     UserContractSignature(
       user_id=current_user.id,
       contract_file_id=contract_file.id,
       rental_id=rental_uuid только для RENTAL_MAIN_CONTRACT, APPENDIX_7_1, APPENDIX_7_2,
       guarantor_relationship_id=guarantor_rel_uuid только для GUARANTOR_*,
       digital_signature=current_user.digital_signature
     )
     db.add(signature), db.commit(), db.refresh(signature).

  9) Обновление полей users (если тип в user_field_mapping):
     user_field_mapping = {
       USER_AGREEMENT: "is_user_agreement",
       CONSENT_TO_DATA_PROCESSING: "is_consent_to_data_processing",
       MAIN_CONTRACT: "is_contract_read"
     }
     setattr(current_user, field_name, True), commit.

  10) Ответ клиенту: UserSignatureResponse (id, user_id, contract_file_id, contract_type, digital_signature, signed_at, rental_id, guarantor_relationship_id).

  11) В фоне (asyncio.create_task):
      - Уведомление в Telegram об успешном подписании.
      - notify_user_status_update(current_user.id).
      - Если договор гаранта — дополнительно notify_user_status_update(guarantor_client_id) (клиент гаранта).


3. АДМИН: POST /contracts/admin/sign (подпись от имени клиента)
--------------------------------------------------------------------------------
Файл: app/contracts/router.py, admin_sign_contract()

  - Только роль ADMIN.
  - Тело: AdminSignContractRequest — user_id (SID клиента), contract_type, rental_id?, guarantor_relationship_id?.
  - Логика та же: поиск ContractFile, проверка дубликата и прав, создание UserContractSignature с user_id = указанный клиент.
  - Обновление user-полей у этого клиента (is_user_agreement, is_consent_to_data_processing, is_contract_read).
  - log_action(..., action="admin_sign_contract_for_user", ...).
  - WebSocket notify_user_status_update для этого user_id (и при гаранте — для client_id).


4. АДМИН: POST /contracts/admin/sign-mechanic (подпись от имени механика)
--------------------------------------------------------------------------------
  - Админ подписывает договор за механика (user_id механика, contract_type, rental_id для договоров аренды).
  - Аналогичная проверка активного ContractFile и дубликата, создание UserContractSignature с user_id = механик.


5. SUPPORT: POST /support/contracts/sign и POST /support/contracts/sign-mechanic
--------------------------------------------------------------------------------
Файл: app/support/contracts/router.py

  - support_sign_contract(): то же по смыслу, что админский sign от имени клиента (support передаёт user_id).
  - support_sign_contract_mechanic(): подпись от имени механика.
  - Логика совпадает с contracts/router.py (активный файл, дубликат, создание подписи, log_action, WebSocket).


6. ГАРАНТ: подписание договоров гаранта
--------------------------------------------------------------------------------
Файл: app/guarantor/router.py, sign_contract() (гарантский эндпоинт)

  - Гарант подписывает GUARANTOR_CONTRACT и GUARANTOR_MAIN_CONTRACT от своего имени.
  - Передаётся contract_type и guarantor_relationship_id (связь Guarantor).
  - Проверка: файл по типу, дубликат, guarantor_rel.guarantor_id == current_user.id.
  - Создание UserContractSignature с guarantor_relationship_id.
  - После подписания обоих договоров гаранта — SMS клиенту (send_guarantor_contract_signed_sms) и уведомление клиенту (WebSocket).


7. ГДЕ ИСПОЛЬЗУЮТСЯ ФЛАГИ ПОДПИСЕЙ
--------------------------------------------------------------------------------
  - reserve_car / аренда: проверка main_contract_signed (UserContractSignature + ContractFile.contract_type == MAIN_CONTRACT).
  - Основной договор аренды: rental_main_contract_signed — подпись RENTAL_MAIN_CONTRACT для данной аренды (rental_id).
  - get_user_me_data (user_data): main_contract_signed, rental_main_contract_signed, appendix_7_1, appendix_7_2, guarantor_contract_signed, guarantor_main_contract_signed — для отображения в приложении и проверок доступа.
  - Гарант: contract_signed (GUARANTOR_CONTRACT), main_contract_signed (GUARANTOR_MAIN_CONTRACT) — для допуска к поручительству.
