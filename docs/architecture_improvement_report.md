# Аналитический отчёт: архитектура и улучшения бэкенда AZV Motors

> На основе документа [business_logic_algorithms.md](business_logic_algorithms.md). Цель — критический анализ архитектуры и конкретные, приоритизированные рекомендации по производительности, надёжности, безопасности, поддерживаемости, observability и эволюции.

---

## 3.1. Общая оценка текущей архитектуры

### Сильные стороны (что уже сделано хорошо)

1. **Финансовая консистентность**
   - Идемпотентность пополнения по `tracking_id` (B14/W3): блокировка `SELECT FOR UPDATE` по существующей транзакции, атомарный `UPDATE users SET wallet_balance = wallet_balance + :amount` предотвращают двойное зачисление.
   - Верификация баланса при завершении аренды (B11): `verify_and_fix_rental_balance` и `recalculate_user_balance_before_rental` приводят баланс и цепочку транзакций в соответствие; вызов из `complete_rental` снижает риск расхождений после сбоев биллинга.

2. **Распределённая блокировка биллинга (B8)**
   - `BillingLock` на Redis гарантирует, что только один инстанс в кластере выполняет `billing_job`; timeout 120 сек и blocking_timeout 0.1 сек — разумный компромисс между дублированием и пропуском. При недоступности Redis — fail-open (один инстанс), что приемлемо для текущего деплоя.

3. **Rate limiting и защита от злоупотреблений**
   - SMS: двухуровневый лимит по номеру (cooldown 60 сек, 5/час) и по IP (10/мин, 50/час, 200/сутки) (A2); graceful degradation на in-memory при отсутствии Redis.
   - GlonassSoft: `RateLimitedHTTPClient` — один запрос в секунду, повтор при 429 (G16), что защищает от блокировки внешнего API.

4. **Снижение нагрузки на БД при телеметрии (G3)**
   - Кэш телеметрии в Redis и `is_significant_change`: обновление `Car` только при смещении ≥1 м, изменении топлива ≥0.5 л или раз в 60 сек — уменьшает число записей в PostgreSQL при опросе каждую секунду.

5. **Observability «на старте»**
   - RequestLoggerMiddleware: trace_id, метод, путь, статус, длительность, user_id, vehicle_id; X-Trace-Id в ответе (M2).
   - HangDetector + HangWatchdog: детекция зависших запросов (>10 сек), запись в ErrorLog с stack traces (M3, M7).
   - PerformanceMonitoringMiddleware: X-Process-Time, предупреждения в Telegram при >3 сек и >10 сек (M6).
   - ErrorLoggerMiddleware: перехват исключений, Telegram + сохранение в ErrorLog (M4).

6. **Защита Swagger** (M5): HTTP Basic Auth для /docs, /redoc, /openapi.json по ENV.

7. **Логика аренды и биллинга**
   - Чёткие переходы статусов (RESERVED → IN_USE → COMPLETED; доставка DELIVERING → DELIVERY_* → IN_USE); флаги `_notification_flags` в `process_rentals_sync` (B9) предотвращают дублирование однотипных push/telegram в рамках одной аренды; ограничение «не более 1 минуты за запуск» для поминутного списания снижает риск массового списания при пропуске job’ов.

### Слабые стороны и архитектурные долги (key pain points)

1. **Router-centric архитектура**
   - Бизнес-логика сосредоточена в хендлерах роутеров (auth, rent, wallet, gps_api, guarantor и т.д.); нет выделенного слоя сервисов/use cases. ORM-запросы и правила разбросаны по роутерам, что затрудняет повторное использование, тестирование и изменение правил без касания HTTP-слоя.

2. **Отсутствие репозиторного слоя**
   - Прямые обращения к `db.query(Model)`, `db.get()`, фильтры и join’ы в роутерах и утилитах; дублирование запросов (например, поиск активной аренды в `get_active_rental`, `get_active_rental_by_car_id` вызывается из многих мест). Усложняет смену источника данных и введение кэширования на уровне доступа к данным.

3. **Один процесс: API + Telegram-бот + APScheduler + HangWatchdog**
   - В одном процессе Uvicorn: REST, WebSocket, Telegram webhook, 8+ периодических задач (в т.ч. опрос телеметрии каждую 1 сек и биллинг каждые 60 сек). Падение или блокировка event loop затрагивает всё; тяжёлые задачи выносятся в `run_in_executor` (G1, B7), но общий ресурс CPU/памяти и блокирующие вызовы (БД, Redis, HTTP) остаются в одном процессе.

4. **Опрос телеметрии каждую 1 секунду (G1)**
   - `scheduler.add_job(check_vehicle_conditions, "interval", seconds=1)`:
     - Постоянная нагрузка на VEHICLES_API_URL (GET /vehicles/?skip=0&limit=100), Redis (load/save_telemetry_batch), БД при значимых изменениях и рассылка WebSocket (`notify_vehicles_list_update`) при любом updated > 0. При росте парка машин и числа подключённых клиентов это станет узким местом по latency и стоимости внешнего API/инфраструктуры.

5. **Биллинг в том же процессе (B7)**
   - `process_rentals_sync` выполняется в `asyncio.to_thread`, но при большом числе активных аренд один длинный проход раз в минуту может создавать пики нагрузки на БД и задержки ответов API. Нет очереди сообщений: при падении процесса в момент биллинга возможна потеря контекста уведомлений/push (списки формируются в памяти и отправляются после выхода из thread).

6. **WebSocket in-memory (N18, G5)**
   - `ConnectionManager` в `app/websocket/manager.py`: подключения хранятся в памяти процесса. При горизонтальном масштабировании (несколько инстансов) клиенты разных инстансов не получают обновления друг друга; sticky sessions или внешний брокер (Redis Pub/Sub, специализированный сервис) не описаны.

7. **CORS и секреты**
   - `allow_origins=["*"]` при `allow_credentials=True` (M1) — небезопасно для production; конфигурация только в коде, без ENV. Системные и тестовые номера (A25) захардкожены в `app/auth/router.py` (SYSTEM_PHONE_NUMBERS, TEST_PHONE_CODES); при утечке репозитория или смене команды это создаёт риски.

8. **Нет централизованного rate limiting на API**
   - Rate limit есть только для SMS (A2). Критические эндпоинты (verify_sms, add_money, reserve-car, start, complete) не защищены общим ограничением по IP/user от brute force и DoS.

9. **Отсутствие распределённого трейсинга и метрик**
   - Есть trace_id в логах и X-Trace-Id в ответе, но нет OpenTelemetry/трассировки между запросами и фоновыми задачами; нет экспорта метрик (Prometheus) по latency, throughput, ошибкам по эндпоинтам и по job’ам — сложно строить SLO и разбирать инциденты.

10. **Финансовые сценарии без явного паттерна Saga/Outbox**
    - Списания и пополнения выполняются в рамках одной транзакции БД; при сбое после commit, но до отправки push/WebSocket пользователь может не увидеть обновление, но данные консистентны. При будущем выводе части операций в очереди (например, уведомления или интеграции с внешними платёжными системами) потребуется Outbox или Saga, чтобы не терять события.

## 3.2. Категории проблем и возможностей

### А. Производительность и масштабирование

**Узкие места**

| Проблема | Где проявляется | Влияние |
|----------|-----------------|---------|
| Опрос телеметрии каждую 1 сек | G1, G2: `check_vehicle_conditions` → `get_last_vehicles_data`, `_update_vehicle_data_sync`, Redis, БД, `notify_vehicles_list_update` | Нагрузка на Vehicles API, Redis, PostgreSQL; при росте парка — рост latency и риска таймаутов |
| Синхронный биллинг в одном потоке | B7, B9: `process_rentals_sync` в `asyncio.to_thread`, все аренды за один проход | Пики нагрузки на БД раз в минуту; при большом N аренд — увеличение времени выполнения job’а и задержки ответов API в том же процессе |
| Отсутствие кэширования для чтения | get_vehicles (G10), get_vehicles_data_for_user (G19), калькулятор стоимости (B21), баланс/транзакции (W4, W5) | Лишние запросы к PostgreSQL на каждый запрос/WebSocket тик; при росте RPS — рост нагрузки на БД |
| Одна БД для операционных и аналитических нагрузок | Отчёты бухгалтерии/финансистов, выписки, экспорт CSV (W7, W8, модуль accountant) | Тяжёлые агрегации конкурируют с транзакционными запросами аренды/кошелька; возможны блокировки и рост p95 latency |

**Решения**

1. **Кэширование телеметрии и списка машин (краткосрочно)**
   - Уже есть кэш «последнего состояния» в Redis для порога значимых изменений (G3). Расширить: для `get_vehicles` и `get_vehicles_data_for_user` отдавать список машин из Redis с TTL 2–5 сек, обновляемый только в `_update_vehicle_data_sync` при `updated > 0`. Это снизит число тяжёлых запросов к БД при частых вызовах GET /vehicles и WebSocket /ws/vehicles/list (G19 — там цикл раз в 1 сек с запросом данных).
   - **Метрики**: снижение QPS к PostgreSQL по таблице cars и связанным; снижение времени ответа GET /vehicles.

2. **Увеличение интервала опроса телеметрии и/или вынос в воркер (среднесрочно)**
   - Поднять интервал с 1 до 2–3 сек (или сделать настраиваемым через ENV); для карты и статусов машин 2–3 сек часто достаточно. Альтернатива: вынести опрос в отдельный воркер (отдельный процесс или контейнер), который только читает Vehicles API, пишет Redis (и при необходимости БД), а основной API читает из Redis и рассылает WebSocket — разделение нагрузки и возможность масштабировать воркер отдельно.
   - **Обоснование**: в документе (G1) указано «интервал 1 сек» и «coalesce=True»; при 100 машинах и десятках клиентов WebSocket каждый тик — десятки вызовов `get_vehicles_data_for_user` и нагрузка на БД. Увеличение интервала или вынос снизят нагрузку на API и Vehicles API.

3. **Асинхронная обработка биллинга через очередь (среднесрочно)**
   - Вместо одного `process_rentals_sync` раз в минуту: job раз в минуту только формирует список rental_id и публикует сообщения в очередь (Redis Streams, RabbitMQ или облачная очередь). Воркеры (отдельные процессы/контейнеры) обрабатывают аренды по одной или батчами: списания, запись транзакций, формирование списков push/telegram/lock_requests. Итоговые списки push/telegram отправляются из воркера или отдельной задачи. Так биллинг не блокирует API и можно масштабировать воркеры; при падении воркера сообщения в очереди не теряются (при использовании acknowledge после обработки).
   - **Промежуточный шаг**: оставить текущий `billing_job` и BillingLock, но разбить `process_rentals_sync` на батчи по N аренд и обрабатывать батчи с небольшой паузой между ними, чтобы снизить пиковую нагрузку на БД.

4. **Репликация БД для чтения (среднесрочно)**
   - Настроить read replica PostgreSQL; направлять тяжёлые read-only запросы (get_vehicles с join’ами, список транзакций с фильтрами, отчёты бухгалтерии) на реплику. Запись (аренда, кошелёк, биллинг) — только в primary. Снизит конкуренцию за ресурсы на primary и улучшит p95 для чтения.

5. **Шардинг БД (долгосрочно)**  
   - При дальнейшем росте данных рассмотреть шардинг по домену (например, телеметрия/снапшоты в отдельную БД или партиционирование больших таблиц по времени). Требует изменения кода доступа к данным и миграций; целесообразно после введения репозиторного слоя.

### Б. Надёжность и отказоустойчивость

**Риски**

| Риск | Где | Последствие |
|------|-----|-------------|
| Потеря контекста при падении во время биллинга | B7: push_notifications, telegram_alerts, lock_requests формируются в памяти в process_rentals_sync, отправка — после выхода из thread | Часть уведомлений или запросов на блокировку двигателя может не быть отправлена; пользователь не получит предупреждение о нулевом балансе |
| Нет retry для критичных операций | ForteBank verify (B14), отправка команд GlonassSoft (G7), Mobizon SMS (A1) | Временные сетевые сбои приводят к 400/500 без повторной попытки; пользователь вынужден повторять действие |
| Единая точка отказа (монолит) | Один процесс: API + бот + scheduler | Падение или OOM убивает весь сервис; деплой и откат затрагивают всё |
| Идемпотентность не везде | add_money — да (tracking_id); start_rental, complete_rental — нет явного idempotency key | Повторная отправка запроса из-за таймаута на клиенте может привести к двойному списанию или двойному завершению при неаккуратной обработке |

**Решения**

1. **Идемпотентность критичных мутаций (краткосрочно)**
   - Ввести опциональный заголовок `Idempotency-Key` для POST /rent/start, POST /rent/complete, POST /rent/cancel и т.п. Сохранять ключ в Redis или в таблице (idempotency_keys: key, user_id, response_status, response_body, created_at) с TTL. При повторном запросе с тем же ключом возвращать сохранённый ответ без повторного выполнения. Это защитит от двойного списания/двойного завершения при таймаутах и повторных нажатиях.
   - **Ссылка на документ**: в B6 (старт аренды) и B10 (завершение) идемпотентность не описана; в B14 (add_money) — есть по tracking_id.

2. **Паттерн Outbox для уведомлений в биллинге (среднесрочно)**
   - В той же транзакции, где обновляются rental и wallet, писать в таблицу `outbox_notifications` (rental_id, user_id, type: push|telegram_lock|…, payload, created_at). Отдельная job или воркер читает outbox и отправляет push/telegram/lock; при успехе помечает запись обработанной (или удаляет). При падении процесса после commit биллинга уведомления не теряются. Можно начать только для lock_requests (блокировка двигателя), как наиболее критичных.

3. **Retry с backoff для внешних вызовов (краткосрочно)**
   - Обернуть вызовы ForteBank (verify_forte_transaction), GlonassSoft (send_open, send_close и т.д.), Mobizon (send_sms_mobizon) в retry с экспоненциальной задержкой (например 1, 2, 4 сек) и ограничением по количеству попыток. Для SMS и платежей — только идемпотентные по бизнесу операции (verify — да, отправка SMS — да; списание — только при идемпотентном ключе). Уменьшит количество 5xx/400 из-за кратковременных сетевых сбоев.

4. **Graceful degradation**
   - Уже частично есть: при недоступности Redis для BillingLock — выполнение биллинга (B8); при недоступности Redis для SMS rate limit — in-memory (A2). Формализовать: для телеметрии при недоступности Vehicles API не падать, а отдавать последние данные из Redis/БД и логировать; при недоступности GlonassSoft для команд возвращать 503 с сообщением «сервис временно недоступен» и предлагать повторить.

5. **Health checks**
   - Расширить /health: проверка подключения к PostgreSQL (SELECT 1), Redis (PING), опционально Vehicles API и GlonassSoft (GET с коротким timeout). Отдельный эндпоинт /ready для Kubernetes (liveness/readiness), который не возвращает 200 при отсутствии БД/Redis — чтобы оркестратор не направлял трафик на неготовый инстанс.

### В. Безопасность

**Проблемы**

| Проблема | Где | Рекомендация |
|----------|-----|---------------|
| CORS `allow_origins=["*"]` | M1 | Заменить на список разрешённых origin из ENV (например CORS_ORIGINS=https://app.azv.kz,https://admin.azv.kz); в production не использовать `*` при credentials |
| Системные/тестовые номера в коде | A25: SYSTEM_PHONE_NUMBERS, TEST_PHONE_CODES в router | Вынести в конфиг (ENV или секрет-менеджер): список номеров и коды; тестовые коды не в коде, а в конфиге для dev/staging |
| Нет rate limiting на критичных эндпоинтах | verify_sms (A3), add_money (B14), reserve-car (B4), start (B6), complete (B10) | Ввести общий middleware или dependency: по IP и/или по user_id ограничить число запросов в минуту (например 10–30 для мутаций на пользователя), чтобы снизить риск brute force и DoS |
| Секреты в коде | Swagger default password (M5), Fernet key для JWT (A5) | Все секреты из ENV или внешнего хранилища (Vault, облачные secrets); без дефолтов в коде для production |
| JWT verify_exp: False (A4) | В документе указано, что срок истечения при верификации отключён | Управление жизнью через TokenRecord и кэш — допустимо, но явно документировать и при необходимости включить verify_exp с большим запасом, чтобы отозванные токены (удаление из БД) не принимались после TTL |

**Решения (приоритет)**

1. **Критично**: ужесточить CORS — список origin из ENV; убрать хардкод системных номеров и тестовых кодов в репозиторий (конфиг/секреты).
2. **Важно**: rate limit по IP и user на POST /auth/verify_sms, POST /rent/add_money, POST /rent/start, POST /rent/complete (и при необходимости reserve-car); использовать Redis с TTL как для SMS.
3. **Желательно**: введение OAuth2/логин через провайдера для админ/механик/финансист — дополнительный слой к JWT по номеру телефона.

### Г. Поддерживаемость и тестируемость

**Проблемы**

- Бизнес-логика в роутерах: сложно покрывать unit-тестами без поднятия HTTP-слоя и БД.
- Дублирование: расчёт цен (calc_required_balance, calculate_total_price) и проверки доступа (get_active_rental, check_user_trips_for_class_access) вызываются из многих мест; изменения правил требуют правок в нескольких файлах.
- Отсутствие явного слоя сервисов и репозиториев: зависимости «роутер → БД/ORM» напрямую, нет абстракций для подмены в тестах.

**Решения**

1. **Выделение сервисного слоя (среднесрочно, эволюционно)**
   - По одному домену за раз (например, аренда): создать `app/rent/services/rental_service.py` с функциями `reserve_car(...)`, `start_rental(...)`, `complete_rental(...)`, принимающими user, car_id, db и т.д. Роутер только парсит запрос, вызывает сервис и формирует ответ. Затем то же для auth (send_sms, verify_sms), wallet (add_money). Без смены контрактов API и без «большого взрыва».
   - **Промежуточный шаг**: вынести из роутеров в отдельные модули (например `app/rent/logic/reserve.py`) чистые функции с явными аргументами; роутеры продолжают вызывать их и работать с db/session. Тесты вызывают эти функции с mock’ами db.

2. **Репозитории (после или параллельно сервисам)**
   - Ввести тонкий слой репозиториев: например `RentalRepository.get_active_by_user(user_id)`, `CarRepository.get_by_id(car_id)`, `WalletRepository.get_balance(user_id)`. Реализация — обёртки над текущими запросами; в тестах подменять репозитории на заглушки. Это упростит введение кэша (например, баланс в Redis с инвалидацией при записи) и чтения с реплики.

3. **Dependency Injection**
   - Использовать FastAPI Depends для внедрения сервисов и репозиториев (фабрики из конфига); в тестах переопределять зависимости. Это уменьшит глобальное состояние и упростит изоляцию тестов.

4. **Contract testing**
   - Для интеграций (ForteBank, GlonassSoft, Mobizon, Vehicles API) ввести контракты (например, Pact или просто сохранённые примеры ответов) и тесты, проверяющие, что наш код корректно обрабатывает ответы и ошибки. Это защитит от поломок при изменении внешних API.

### Д. Observability

**Проблемы**

- Логи есть (в т.ч. структурированные при LOG_FORMAT=json), но нет метрик по эндпоинтам (latency percentiles, RPS, error rate) и по job’ам (billing_job duration, count of processed rentals; check_vehicle_conditions duration, updated_count).
- Нет распределённого трейсинга: trace_id есть в рамках одного запроса, но не связывается с фоновыми задачами (биллинг, телеметрия) и с внешними вызовами (ForteBank, GlonassSoft).

**Решения**

1. **Prometheus-метрики (краткосрочно)**
   - Экспорт метрик в формате Prometheus (эндпоинт /metrics): счётчики запросов по методу и пути, гистограммы длительности запросов по эндпоинту, счётчики ошибок 4xx/5xx. Отдельно метрики для job’ов: billing_job_duration_seconds, billing_rentals_processed_total; check_vehicle_conditions_duration_seconds, telemetry_updated_count. Это позволит строить дашборды и алерты по SLO (например, p95 latency, error rate).

2. **Структурированное логирование**
   - Уже используется extra в RequestLogger (M2). Унифицировать ключи (trace_id, user_id, rental_id, car_id, duration_ms) и при необходимости добавлять их в логи из сервисов и billing’а, чтобы по trace_id собирать полный путь запроса и фоновой задачи в логах (например, в Loki/ELK).

3. **OpenTelemetry (среднесрочно)**
   - Внедрить трейсинг: автоматическая инструментация FastAPI (span на запрос), ручные span’ы для billing_job, check_vehicle_conditions, вызовов ForteBank/GlonassSoft/Mobizon. Экспорт в Jaeger или облачный трейсер. Связь trace_id запроса с последующими операциями (если клиент ждёт ответа, а внутри вызывается биллинг — не обязательно; но для фоновых задач общий trace_id по job run упростит разбор инцидентов).

### Е. Эволюция к микросервисам

**Оценка необходимости**

- Сейчас монолит оправдан: небольшая команда, один продукт, проще деплой и отладка. Разбиение на микросервисы имеет смысл при: необходимости независимого масштабирования частей (телеметрия vs API vs биллинг), разных циклах релизов или командах, жёстких требованиях к изоляции (например, PCI для платежей).
- По документу естественные кандидаты на выделение:
  - **Телеметрия/GPS**: опрос Vehicles API, обновление БД/Redis, детекция заправки, уведомления по телеметрии; высокая частота и нагрузка (G1, G2). Выделение в отдельный сервис позволит масштабировать его по числу машин и снизить влияние на основной API.
  - **Биллинг**: периодическая обработка аренд, списания, push/telegram/lock (B7, B9). Выделение в воркер(ы) с очередью (см. п. А) даёт масштабируемость и отказоустойчивость без обязательного переименования в «микросервис».
  - **Платежи/кошелёк**: верификация ForteBank, пополнение, списания (B14, W1–W3). Выделение имеет смысл при росте регуляторных требований или при появлении нескольких платёжных провайдеров; можно отложить.
  - **Поддержка (Telegram-бот)**: уже отдельный контур по логике (support_reply_states in-memory, модуль support); при горизонтальном масштабировании API in-memory состояние бота не масштабируется. Вариант: вынести бота в отдельный процесс с тем же кодом или в отдельный сервис с Redis/БД для состояния.

**Стратегия (если пойдём по пути выделения)**

- **Strangler fig**: новый функционал телеметрии или биллинга реализовывать в отдельном сервисе/воркере; старый код в монолите постепенно отключать после переключения трафика/данных.
- **API Gateway**: при наличии нескольких сервисов ввести единую точку входа (Kong, AWS ALB + path-based, или минимальный BFF), маршрутизация по путям; CORS и rate limit централизовать в gateway.
- **Взаимодействие**: синхронное (HTTP) для запросов «запросил — получил ответ» (например, API → сервис телеметрии за последним состоянием машины); асинхронное (очередь) для биллинга, уведомлений, событий (аренда начата/завершена) — чтобы развязать производительность и не терять события при сбоях.

## 3.4. Заключение

**Резюме основных рекомендаций**

1. **Архитектура**: Сохранять монолит как основу, но эволюционно выделять сервисный слой и репозитории; тяжёлые и частые задачи (телеметрия каждую секунду, биллинг раз в минуту) выносить в воркеры с очередями и по возможности в отдельные процессы/контейнеры для масштабирования и отказоустойчивости.

2. **Производительность**: Снизить нагрузку от телеметрии (интервал 2–3 сек или отдельный воркер); кэшировать список машин в Redis; направлять read-only нагрузку на реплику; в перспективе — биллинг через очередь с воркерами.

3. **Надёжность**: Ввести идемпотентность для старта/завершения/отмены аренды; Outbox для уведомлений биллинга (как минимум lock); retry с backoff для внешних API; явные health checks для БД и Redis.

4. **Безопасность**: Ужесточить CORS (список origin из ENV); убрать хардкод системных и тестовых номеров; ввести rate limiting на критичные эндпоинты; хранить все секреты во внешнем конфиге/секретах.

5. **Поддерживаемость**: Выделить сервисы аренды и auth из роутеров; ввести репозитории и DI; унифицировать структурированные логи и контракты для внешних API.

6. **Observability**: Ввести Prometheus-метрики по запросам и job’ам; добавить распределённый трейсинг (OpenTelemetry) для запросов и ключевых фоновых операций.

Приоритет для стартапа: быстрые победы по безопасности (CORS, конфиг, rate limit) и наблюдаемости (метрики, health), затем кэш телеметрии и интервал опроса, затем идемпотентность и Outbox для биллинга, и параллельно — выделение сервисного слоя по доменам без остановки прода.
