# Анализ Sequence: Полный цикл аренды

Детальный анализ sequence-диаграммы жизненного цикла аренды: от бронирования до завершения и пост-оплаты.

---

## 2.1. Общая характеристика

### Бизнес-сценарии

- **Бронирование** — создание резерва машины и проверка баланса кошелька.
- **Старт аренды** — верификация пользователя (selfie), разблокировка дверей через GPS, переход аренды в статус IN_USE.
- **Биллинг** — периодическое списание по минутам с кошелька во время поездки.
- **Завершение** — блокировка дверей и двигателя, фиксация пробега/топлива, расчёт итоговой стоимости, переход в COMPLETED.
- **Пост-оплата** — пополнение кошелька через ForteBank при отрицательном балансе и верификация транзакции.

### Компоненты

| Участник | Роль |
|----------|------|
| **Пользователь (U)** | Инициатор запросов через мобильное приложение. |
| **FastAPI (API)** | Единая точка входа: маршрутизация, бизнес-логика, вызовы внешних сервисов и БД. |
| **PostgreSQL (DB)** | Хранение RentalHistory, Car, User, WalletTransaction, блокировки. |
| **GlonassSoft (GPS)** | Управление замками и двигателем (разблокировка/блокировка). |
| **ForteBank (FB)** | Приём платежей и проверка статуса транзакции по `tracking_id`. |
| **BillingJob (BJ)** | Фоновый процесс: выбор активных аренд и списание по минутам. |

## 2.2. Детальный пошаговый разбор

### Фаза 1: Бронирование

| Шаг | Инициатор | Получатель | Данные / Действие | Изменения БД | Внешние вызовы |
|-----|-----------|------------|-------------------|--------------|----------------|
| 1 | U | API | `POST /rent/booking/advanced` (car_id, опционально время старта) | — | — |
| 2 | API | DB | Проверить баланс кошелька пользователя | SELECT users.wallet_balance, расчёт требуемой суммы (calc_required_balance) | — |
| 3 | API | DB | Создать запись RentalHistory (status=RESERVED или SCHEDULED для отложенного бронирования) | INSERT RentalHistory | — |

| 5 | API | U | Ответ с `booking_id` | — | — |

**Примечание:** В коде эндпоинты называются иначе: `POST /rent/reserve-car/{car_id}` (мгновенное) и `POST /rent/advance-booking` (отложенное). При недостатке баланса возвращается 402; при включённой логике может списываться rebooking_fee.

| Шаг | Инициатор | Получатель | Данные / Действие | Изменения БД | Внешние вызовы |
|-----|-----------|------------|-------------------|--------------|----------------|
| 1 | U | API | `POST /rent/start-rental` (car_id, selfie) | — | — |
| 2 | API | API | Сравнение selfie с эталоном (если включено) | — | — |

**В коде:** Старт — `POST /rent/start/{car_id}`. Сначала проверка баланса на base + open_fee + delivery_fee, запись транзакций (RENT_BASE_CHARGE и т.д.), затем вызов GPS. Selfie проверяется в том же запросе; при неудаче поток прерывается.

### Фаза 3: Поездка (биллинг)

| Шаг | Инициатор | Получатель | Данные / Действие | Изменения БД | Внешние вызовы |
|-----|-----------|------------|-------------------|--------------|----------------|
| 1 | BJ | DB | Найти аренды в статусе IN_USE (и др. биллинговые статусы) | SELECT RentalHistory с блокировкой (BillingLock) | — |
| 2 | BJ | DB | Списать price_per_minute с кошелька пользователя | INSERT WalletTransaction (RENT_MINUTE_CHARGE), UPDATE rental, UPDATE users.wallet_balance | — |
| 3 | — | — | Повтор каждые ~60 секунд | — | — |

**В коде:** BillingJob обрабатывает до одной минуты за запуск; при балансе ≤ 0 — пуш пользователю, затем после задержки (например 10 мин) — добавление в lock_requests и отправка команды блокировки двигателя через GPS. На диаграмме этот сценарий «отрицательный баланс во время поездки» не отражён.

### Фаза 4: Завершение аренды

| Шаг | Инициатор | Получатель | Данные / Действие | Изменения БД | Внешние вызовы |
|-----|-----------|------------|-------------------|--------------|----------------|
| 1 | U | API | `POST /rent/complete-rental` (photos, опционально fuel_after, mileage_after) | — | — |
| 2 | API | GPS | Заблокировать двери и двигатель | — | GlonassSoft API (lock) |
| 3 | API | DB | Записать photos_after, fuel_after, mileage_after | UPDATE RentalHistory | — |
| 4 | API | DB | Рассчитать total_price (base + overtime + distance + fuel) и при необходимости скорректировать баланс | UPDATE RentalHistory (total_price, base_price, overtime_fee, …); verify_and_fix_rental_balance; WalletTransaction при доплате | — |
| 5 | API | DB | RentalHistory → COMPLETED, end_time | UPDATE RentalHistory | — |
| 6 | API | DB | Car → свободен | UPDATE Car SET status=PENDING (в коде не FREE) | — |
| 7 | API | U | Ответ с чеком (total_price, breakdown) | — | — |

**Формула итоговой стоимости:** В коде используется `calculate_total_price` / пересчёт при завершении: base_price + overtime_fee + distance_fee + топливо (разница fuel_before − fuel_after по тарифу). Источники: минуты/пробег из RentalHistory и Car, топливо — из переданных и сохранённых значений.

### Фаза 5: Пост-оплата (при отрицательном балансе)

| Шаг | Инициатор | Получатель | Данные / Действие | Изменения БД | Внешние вызовы |
|-----|-----------|------------|-------------------|--------------|----------------|
| 1 | U | FB | Пополнение через ForteBank (вне приложения/интеграции) | — | ForteBank (платёж) |
| 2 | FB | U | Возврат `tracking_id` | — | — |
| 3 | U | API | `POST /rent/verify-forte-transaction` с tracking_id | — | — |
| 4 | API | FB | Проверка статуса платежа по tracking_id | — | ForteBank API (verify_forte_transaction) |
| 5 | API | DB | Зачислить сумму на wallet_balance, записать транзакцию | INSERT WalletTransaction (DEPOSIT, tracking_id); UPDATE users.wallet_balance | — |

**В коде:** Эндпоинт — `POST /rent/add_money` (amount, tracking_id). Идемпотентность: поиск существующей WalletTransaction с тем же tracking_id под `SELECT ... FOR UPDATE`; при наличии — ответ «Транзакция уже была обработана» без повторного зачисления. Сумма может быть переопределена значением от ForteBank (verified_amount).

## 2.3. Анализ критических точек

### Проверка баланса при бронировании

- **Что проверяется:** Баланс кошелька пользователя (`users.wallet_balance`) против требуемой суммы.
- **Требуемая сумма:** Рассчитывается через `calc_required_balance` (резерв на время бронирования + предоплата и т.д.). При недостатке средств возвращается 402.
- **Уточнение для ТЗ:** Явно указать, что проверяется «достаточность баланса для минимального резерва и предоплаты по правилам тарифа».

### Selfie-верификация

- **Влияние на поток:** Опциональна (настройка). Если включена — проверка в рамках `POST /rent/start/{car_id}`; при неудаче запрос завершается ошибкой, аренда не переводится в IN_USE и двери не разблокируются.
- **Пробелы на диаграмме:** Нет ветки «selfie не прошёл» (alt) и нет таймаута ожидания selfie.

### Биллинг

- **Атомарность:** Списание минуты выполняется в контексте BillingLock и транзакции БД (запись WalletTransaction + обновление rental и users.wallet_balance).
- **Отрицательный баланс во время поездки:** В коде предусмотрено: пуш пользователю, затем по истечении периода — блокировка двигателя через GPS. На диаграмме этот сценарий не отражён — рекомендуется добавить opt/alt «при балансе ≤ 0».

### Завершение: расчёт итоговой стоимости

- **Формула:** total_price = base_price + overtime_fee + distance_fee + плата за топливо (при fuel_after < fuel_before).
- **Источники данных:** Минуты и пробег — из RentalHistory и связанных сущностей; топливо — переданные при завершении и сохранённые fuel_before/fuel_after.
- **Корректировка баланса:** Вызов `verify_and_fix_rental_balance` при завершении гарантирует, что с кошелька списана итоговая сумма (при необходимости — доплата).

### Пост-оплата и идемпотентность

- **Взаимодействие с ForteBank:** Проверка транзакции по `tracking_id` через `verify_forte_transaction(tracking_id)`.
- **Идемпотентность:** Повторная отправка того же `tracking_id` не приводит к двойному зачислению: проверка существующей WalletTransaction с этим tracking_id под блокировкой (FOR UPDATE); при совпадении возвращается сообщение об уже обработанной транзакции.

## 2.4. Соответствие кодовой базе

### Соответствие по эндпоинтам

| Диаграмма | Код (app/rent/router.py и др.) |
|-----------|--------------------------------|
| `POST /rent/booking/advanced` | `POST /rent/advance-booking` или `POST /rent/reserve-car/{car_id}` |
| `POST /rent/start-rental` | `POST /rent/start/{car_id}` |
| `POST /rent/complete-rental` | `POST /rent/complete` |
| `POST /rent/verify-forte-transaction` | Логика внутри `POST /rent/add_money` (tracking_id + verify_forte_transaction) |

### Соответствие алгоритмам (docs/business_logic_algorithms.md)

- **Бронирование (B4, reserve-car / advance-booking):** Проверка баланса, создание RentalHistory (RESERVED/SCHEDULED), обновление Car — совпадает. На диаграмме не отражены: rebooking_fee, различие advance vs instant.
- **Старт (B5, start):** Баланс, транзакции (base + open_fee + delivery_fee), GPS unlock, IN_USE, photos_before — совпадает. Selfie — в коде есть, на диаграмме помечено как опция.
- **Биллинг (B7–B8, billing_job, process_rentals_sync):** Минутное списание, блокировки, WalletTransaction — совпадает. В коде есть обработка нулевого/отрицательного баланса (пуш, lock_engine) — на диаграмме отсутствует.
- **Завершение (B9–B10, complete):** GPS lock, photos_after, fuel_after, mileage_after, пересчёт total_price, verify_and_fix_rental_balance, COMPLETED — совпадает. В коде Car переводится в PENDING, а не FREE — расхождение в диаграмме.
- **Пост-оплата:** add_money + verify_forte_transaction + идемпотентность по tracking_id — совпадает; имя эндпоинта на диаграмме условное.

### Рекомендации для ТЗ

- Унифицировать названия эндпоинтов с реальными путями.
- Указать статус автомобиля после завершения как PENDING (или уточнить переход PENDING → FREE в отдельном процессе).

## 2.5. Полнота и пробелы

### Отсутствующие шаги и сценарии

- **Отмена бронирования** — снятие RESERVED/SCHEDULED, возврат средств/резерва, обновление Car.
- **Продление аренды** — решение о продлении, влияние на биллинг и total_price.
- **Штрафы и доп. платежи** — повреждения, штрафы за нарушение правил; на диаграмме не показаны.
- **Плата за ожидание (RESERVED)** — списание за время удержания машины в резерве (если есть в коде).
- **Доставка машины (DELIVERING)** — отдельный статус и биллинг до старта поездки.
- **Ошибки и таймауты:**  
  - Ответ GPS при ошибке разблокировки/блокировки (retry, отмена старта/завершения).  
  - Таймаут при ожидании selfie (если приложение ждёт снимок).
- **Отрицательный баланс во время поездки** — пуш, задержка, блокировка двигателя (lock_engine) через GPS.
- **Разделение типов бронирования** — advance-booking (SCHEDULED) vs reserve-car (RESERVED) и разное поведение при старте.

### Альтернативные сценарии, не показанные на диаграмме

- Старт не выполнен (selfie failed / недостаточно средств / машина уже занята).
- Завершение с отрицательным балансом и последующее пополнение (показано только общее «opt»).
- Двойной вызов complete или add_money с тем же tracking_id (идемпотентность).
- Одновременный биллинг по одной аренде (блокировки BillingLock).

## 2.6. Рекомендации по улучшению диаграммы для ТЗ

1. **Имена эндпоинтов**  
   Привести в соответствие с API:  
   `POST /rent/advance-booking` или `POST /rent/reserve-car/{car_id}`,  
   `POST /rent/start/{car_id}`,  
   `POST /rent/complete`,  
   `POST /rent/add_money` (с параметром tracking_id).

2. **Статус Car после завершения**  
   Указать переход в PENDING (или описать отдельный переход PENDING → FREE).

3. **Биллинг при нуле/отрицательном балансе**  
   Добавить opt/alt в цикле биллинга: «Если баланс ≤ 0: уведомление пользователю, затем (через N мин) команда блокировки двигателя через GPS».

4. **Selfie**  
   Добавить alt при включённой верификации: «Selfie не прошёл → ответ с ошибкой, аренда остаётся RESERVED».

5. **Критические ошибки**  
   Добавить opt/alt для сбоев GPS (разблокировка/блокировка): повтор запроса или отмена операции с понятным статусом аренды.

6. **Отмена и продление**  
   При необходимости ТЗ: отдельные фрагменты для отмены бронирования и продления аренды.

7. **Пост-оплата**  
   Подписать эндпоинт как `POST /rent/add_money` и указать в комментарии идемпотентность по tracking_id.

8. **Участник BillingJob**  
   Оставить явно, подчеркнуть периодичность (например, «каждые 60 с») и использование блокировок при списании.

После этих правок диаграмма будет лучше соответствовать коду и пригодна для включения в ТЗ как описание полного цикла аренды и пост-оплаты.
