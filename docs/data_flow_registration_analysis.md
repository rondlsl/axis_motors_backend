# Анализ Data Flow: Поток регистрации пользователя

Документ подготовлен в рамках анализа архитектуры backend проекта **AZV Motors** (каршеринг в Алматы). Разбирается Data Flow диаграмма процесса регистрации — от ввода телефона до получения полного доступа (роль USER) — и даётся критическая оценка с учётом реализации в коде (auth, financier, mvd, push).

---

## 1. Общее описание

На диаграмме показан **сквозной сценарий регистрации и верификации** нового пользователя:

1. **SMS-верификация** — пользователь вводит номер телефона; backend отправляет SMS-код (Mobizon) и создаёт или обновляет запись пользователя с ролью **CLIENT**. После ввода кода выдаётся JWT, клиент получает доступ к защищённым эндпоинтам.
2. **Загрузка документов** — авторизованный пользователь загружает фото документов (ID, права, селфи и при необходимости справки); файлы сохраняются в MinIO, в БД создаётся или обновляется заявка (**Application**), роль пользователя переводится в **PENDINGTOFIRST** (ожидание проверки финансистом).
3. **Проверка финансистом** — в админ-панели финансист одобряет или отклоняет заявку. При одобрении: **financier_status = APPROVED**, роль пользователя → **PENDINGTOSECOND**. При отклонении: **financier_status = REJECTED**, роль пользователя → одна из **REJECTFIRST*** (в зависимости от причины); на диаграмме условно показан возврат к «DB3» (ожидание повторной загрузки документов только для части отказов).
4. **Проверка МВД** — МВД одобряет или отклоняет заявку (доступна только после одобрения финансистом или при наличии активного гаранта для REJECTFIRST). При одобрении: **mvd_status = APPROVED**, роль → **USER** (полный доступ к аренде). При отклонении: **mvd_status = REJECTED**, роль → **REJECTSECOND**, пользователь деактивируется (is_active = False); повторная регистрация по тому же номеру блокируется.
5. **Push-уведомления** — при переходе в PENDINGTOSECOND и при присвоении роли USER пользователю отправляются push (FCM/Expo); дополнительно после загрузки документов отправляется уведомление «документы загружены».

Покрыты основные шаги до получения роли USER. **Не отражены явно**: вариации отказа финансиста (документы / сертификаты / финансовые причины), поручительство (REJECTFIRST + гарант), повторная загрузка документов после REJECTFIRSTDOC/REJECTFIRSTCERT, блокировка REJECTSECOND при send_sms/verify_sms, email-верификация, rate limiting SMS.

## 2. Участники

| Участник | Роль в потоке |
|----------|----------------|
| **Клиент (мобильное приложение)** | Ввод телефона, ввод SMS-кода, загрузка документов и личных данных; отображение статуса заявки и получение push. |
| **Backend API** | Обработка POST /auth/send_sms, /auth/verify_sms, /auth/upload-documents; создание/обновление User и Application; вызов внешних сервисов (Mobizon, MinIO); не выполняет проверку финансистом/МВД — только админ-панель. |
| **Админ-панель (финансист)** | Просмотр заявок со статусом financier_status = PENDING; одобрение (approve) или отклонение (reject) с указанием причины/типа; смена роли пользователя и статусов заявки. |
| **Админ-панель (МВД)** | Просмотр заявок, одобренных финансистом (или REJECTFIRST с активным гарантом); одобрение или отклонение; при одобрении — роль USER, при отклонении — REJECTSECOND и деактивация. |
| **Mobizon (SMS)** | Отправка SMS с кодом подтверждения на номер телефона пользователя. |
| **MinIO** | Хранение загруженных файлов (фото документов, селфи); возврат URL для сохранения в полях User (id_front_url, id_back_url, drivers_license_url и т.д.). |
| **FCM / Expo Push** | Отправка push-уведомлений пользователю: «документы загружены», «одобрено финансистом», «одобрено МВД», «проверка пройдена». В проекте используется Expo Push API. |
| **PostgreSQL** | Хранение User (в т.ч. role, last_sms_code, sms_code_valid_until, документы URL), Application (financier_status, mvd_status, reason, даты и ID сотрудников). |

## 3. Пошаговый разбор

### 3.1. SMS-верификация

#### Запрос кода (POST /auth/send_sms)

| Аспект | Описание |
|--------|----------|
| **Действие клиента** | Ввод номера телефона; для нового пользователя — first_name, last_name (обязательно), middle_name, email (опционально). |
| **Вызов API** | `POST /auth/send_sms/` (тело: phone_number, first_name, last_name, …). |
| **Внешние сервисы** | **Mobizon** — отправка SMS с кодом (если не тестовый номер и SMS_TOKEN ≠ "6666"). При указании email — дополнительно отправка кода на email (SMTP). |
| **БД** | **Чтение**: проверка блокировки по номеру (REJECTSECOND → 403); поиск активного пользователя по phone_number. **Запись**: при отсутствии активного — восстановление неактивного (is_active = True) или создание нового User (обязательны first_name, last_name); у всех — обновление last_sms_code, sms_code_valid_until. Роль нового пользователя по умолчанию **CLIENT** (default в модели User). |
| **Ограничения** | Rate limiting: по номеру 60 сек cooldown, макс 5 SMS/час; по IP — макс 10/мин, 50/час, 200/сутки. Системные/тестовые номера — фиксированные коды без отправки в сеть. |

#### Ввод кода и выдача JWT (POST /auth/verify_sms)

| Аспект | Описание |
|--------|----------|
| **Действие клиента** | Ввод номера и SMS-кода. |
| **Вызов API** | `POST /auth/verify_sms/` (тело: phone_number, sms_code). |
| **БД** | **Чтение**: поиск пользователя по phone_number, last_sms_code, sms_code_valid_until > now (для обычных пользователей); для системных — без проверки срока. **Запись**: last_activity_at; создание записей TokenRecord (access + refresh). Проверки: REJECTSECOND → 403; is_blocked, is_deleted → 403. |
| **Выходные данные** | Access и refresh JWT, данные пользователя (в т.ч. role). Роль при верификации **не меняется** — остаётся CLIENT до загрузки документов. Связывание заявок поручителя (GuarantorRequest) по номеру телефона; при первом входе — генерация digital_signature. |
| **Уточнение по диаграмме** | Связь «S2 → DB2» и «JWT → U3» корректна: после успешной верификации клиент получает токены и может вызывать POST /auth/upload-documents. |

### 3.2. Загрузка документов

| Аспект | Описание |
|--------|----------|
| **Действие клиента** | Загрузка обязательных файлов (id_front, id_back, drivers_license, selfie_with_license, selfie) и при необходимости справок (psych_neurology_certificate, pension_contributions_certificate для граждан РК); передача данных формы (ФИО, дата рождения, ИИН/паспорт, сроки действия документов, is_citizen_kz, email). |
| **Вызов API** | `POST /auth/upload-documents/` (multipart/form-data + JWT). |
| **Внешние сервисы** | **MinIO** — сохранение каждого файла (save_file_to_minio), возврат URL; URL записываются в поля User (id_front_url, id_back_url, drivers_license_url, selfie_url, selfie_with_license_url, psych_neurology_certificate_url, pension_contributions_certificate_url). |
| **БД** | **Чтение**: текущий пользователь (из JWT), существующая заявка Application по user_id. **Запись**: обновление полей User (ФИО, birth_date, iin, passport_number, даты документов, is_citizen_kz, upload_document_at, documents_verified = True и др.); при отсутствии заявки — создание Application (financier_status = PENDING, mvd_status = PENDING); при наличии заявки и роли REJECTFIRSTDOC/REJECTFIRSTCERT — сброс financier_status/mvd_status в PENDING и reason; **роль пользователя → PENDINGTOFIRST** (если заявка не была полностью одобрена). |
| **Уведомления** | Push «documents_uploaded» (send_localized_notification_to_user_async); WebSocket notify_user_status_update(user_id). |
| **Уточнение по диаграмме** | На диаграмме «S3 → DB3» отображает смену роли на PENDINGTOFIRST и сохранение документов; создание/обновление Application (DB4/DB6) логически происходит в этом же шаге (заявка создаётся при первой загрузке документов). |

### 3.3. Проверка финансистом

| Аспект | Описание |
|--------|----------|
| **Действие** | В админ-панели финансист открывает заявки со статусом financier_status = PENDING и выполняет одобрение или отклонение. |
| **Вызов API** | Эндпоинты модуля financier (например POST approve/{application_id}, POST reject/{application_id}) с указанием причины/типа при отклонении. |
| **БД при одобрении** | Application: financier_status = **APPROVED**, financier_approved_at, financier_user_id. User: role = **PENDINGTOSECOND**, auto_class = выбранные классы авто. Обновление auto_class у связанных гарантов. |
| **БД при отклонении** | Application: financier_status = **REJECTED**, financier_rejected_at, financier_user_id, reason. User: role = **REJECTFIRST** (финансовые причины), **REJECTFIRSTDOC** (документы нечитаемы) или **REJECTFIRSTCERT** (нет справок для граждан РК); auto_class = None. Отмена заявок поручителей (cancel_guarantor_requests_on_rejection). |
| **Уведомления** | При одобрении — push «financier_approve» / «application_approved_financier» (с auto_class); WebSocket для user и гарантов. При отклонении — push «financier_reject_documents» / «financier_reject_certificates» / «financier_reject_financial»; WebSocket для user и гарантов. |
| **Уточнение по диаграмме** | «DB4 → approved → DB5» и «DB4 → rejected → DB3»: при отклонении в коде роль становится REJECTFIRST* (не «остаётся» PENDINGTOFIRST). Возврат к «ожиданию загрузки документов» возможен только при повторной загрузке документов пользователем (тогда для REJECTFIRSTDOC/REJECTFIRSTCERT сбрасываются статусы заявки и роль снова PENDINGTOFIRST). На диаграмме DB3 лучше трактовать как «пользователь в ожидании/отклонён на первом этапе» с возможностью повторной загрузки. |

---

### 3.4. Проверка МВД

| Аспект | Описание |
|--------|----------|
| **Действие** | МВД в админ-панели рассматривает заявки с financier_status = APPROVED (или REJECTFIRST с активным принятым гарантом) и mvd_status = PENDING. |
| **Вызов API** | Эндпоинты модуля MVD (POST approve/{application_id}, POST reject/{application_id}). |
| **БД при одобрении** | Application: mvd_status = **APPROVED**, mvd_approved_at, mvd_user_id. User: если role ≠ REJECTFIRST — role = **USER**. (Для REJECTFIRST с гарантом роль остаётся REJECTFIRST, но доступ к аренде может быть разрешён при наличии гаранта.) |
| **БД при отклонении** | Application: mvd_status = **REJECTED**, mvd_rejected_at, mvd_user_id, reason. User: role = **REJECTSECOND**, **is_active = False**; отмена заявок поручителей. |
| **Уведомления** | При одобрении — push «mvd_approve» / «application_approved_mvd», при наличии токенов — «verification_passed»; WebSocket notify_user_status_update. При отклонении — push «mvd_reject» (в коде). |
| **Уточнение по диаграмме** | «DB6 → approved → DB7» и «DB6 → rejected → DB3»: при отклонении МВД пользователь переходит в **REJECTSECOND** и деактивируется, а не возвращается к PENDINGTOFIRST. Повторная регистрация по тому же номеру при REJECTSECOND блокируется в send_sms и verify_sms (403). На диаграмме для МВД reject корректнее показать переход в отдельное состояние «REJECTSECOND» (без доступа к приложению). |

### 3.5. Push-уведомления

| Момент | Содержание |
|--------|------------|
| После загрузки документов | «Документы загружены» (documents_uploaded) — отправка из upload_documents. |
| Одобрение финансистом | «Заявка одобрена финансистом», доступ к классам авто (financier_approve / application_approved_financier). |
| Одобрение МВД | «Заявка одобрена МВД» (mvd_approve), «Проверка пройдена» (verification_passed). |
| Отклонение финансистом | В зависимости от причины: документы нечитаемы, нет сертификатов, финансовый отказ (financier_reject_*). |
| Отклонение МВД | Уведомление об отклонении (mvd_reject). |

На диаграмме блоки «DB5 → FCM_REG» и «DB7 → FCM_REG» соответствуют отправке push при переходе в PENDINGTOSECOND и USER. Стоит дополнительно показать push после загрузки документов (из S3 → FCM_REG или отдельной связью).

## 4. Анализ статусной модели

### 4.1. Роли пользователя (User.role)

| Роль | Описание | Когда устанавливается |
|------|----------|------------------------|
| **CLIENT** | Пользователь прошёл SMS-верификацию, документы не загружал или ещё не обработаны. | По умолчанию при создании User; после send_sms для нового пользователя. |
| **PENDINGTOFIRST** | Документы загружены, заявка на рассмотрении у финансиста. | После успешного upload_documents (или повторной загрузки после REJECTFIRSTDOC/REJECTFIRSTCERT). |
| **PENDINGTOSECOND** | Финансист одобрил, заявка на рассмотрении у МВД. | После одобрения финансистом. |
| **USER** | Полная верификация пройдена, доступ к аренде (при выполнении остальных условий). | После одобрения МВД (если до этого role ≠ REJECTFIRST). |
| **REJECTFIRST** | Отказ финансиста по финансовым причинам; возможен доступ через поручителя. | При отклонении финансистом с reason_type = financial. |
| **REJECTFIRSTDOC** | Отказ финансиста: документы нечитаемы; можно загрузить документы заново. | При отклонении с reason_type = documents. |
| **REJECTFIRSTCERT** | Отказ финансиста: отсутствуют справки для граждан РК; можно дозагрузить. | При отклонении с reason_type = certificates. |
| **REJECTSECOND** | Окончательный отказ МВД; вход по этому номеру заблокирован. | При отклонении МВД; user.is_active = False. |

Остальные роли (ADMIN, MECHANIC, FINANCIER, MVD, GARANT, SUPPORT, DRIVER, ACCOUNTANT) в потоке регистрации не участвуют.

### 4.2. Статусы заявки (Application)

| Поле | Значения | Описание |
|------|----------|----------|
| **financier_status** | PENDING, APPROVED, REJECTED | Рассмотрение финансистом. |
| **mvd_status** | PENDING, APPROVED, REJECTED | Рассмотрение МВД (после одобрения финансистом или при REJECTFIRST с гарантом). |

Дополнительно в Application хранятся: financier_approved_at, financier_rejected_at, financier_user_id; mvd_approved_at, mvd_rejected_at, mvd_user_id; reason (общая причина отклонения).

### 4.3. Переходы на диаграмме

- **Отражены**: CLIENT (после send_sms); проверка кода → JWT; PENDINGTOFIRST после загрузки документов; financier approve → PENDINGTOSECOND; financier reject → возврат к «ожиданию» (уточнено выше); mvd approve → USER; mvd reject → условный возврат к DB3 (в реальности REJECTSECOND).
- **Не отражены явно**: различие REJECTFIRST / REJECTFIRSTDOC / REJECTFIRSTCERT; переход REJECTSECOND (и блокировка входа); сценарий «запрос на повторную проверку» (финансист возвращает заявку в PENDING с уведомлением пользователю); поручительство (REJECTFIRST + гарант → рассмотрение МВД).

## 5. Критическая оценка

### 5.1. Полнота

- **Присутствуют**: SMS-верификация, выдача JWT, загрузка документов, MinIO, смена роли на PENDINGTOFIRST, проверка финансистом и МВД, переходы approved/rejected, push при PENDINGTOSECOND и USER.
- **Отсутствуют или упрощены**:
  - **Повторная загрузка документов** после REJECTFIRSTDOC / REJECTFIRSTCERT: сброс financier_status/mvd_status в PENDING, роль снова PENDINGTOFIRST.
  - **Поручительство**: пользователь с REJECTFIRST может привлечь гаранта; заявка с активным гарантом может быть рассмотрена МВД (одобрение → доступ к аренде при сохранении роли REJECTFIRST).
  - **Блокировка REJECTSECOND**: при send_sms и verify_sms возврат 403, повторная регистрация по тому же номеру невозможна.
  - **Email-верификация**: при указании email в upload_documents может быть запрошена верификация; не влияет на смену роли, но влияет на данные в ответе.
  - **Rate limiting SMS** и **системные/тестовые номера** (без реальной отправки SMS).
  - **Запрос на повторную проверку** (финансист возвращает заявку в PENDING, пользователь получает push «documents_recheck_required» / «financier_request_recheck»).

### 5.2. Соответствие коду

- **Совпадения**: эндпоинты POST /auth/send_sms, /auth/verify_sms, /auth/upload-documents; создание User с ролью CLIENT; смена роли на PENDINGTOFIRST после загрузки документов; создание/обновление Application; financier_status/mvd_status PENDING → APPROVED/REJECTED; смена роли на PENDINGTOSECOND при одобрении финансистом и на USER при одобрении МВД; push и WebSocket при смене статусов.
- **Расхождения**:
  - При отклонении финансистом роль не «остаётся» PENDINGTOFIRST, а переходит в REJECTFIRST / REJECTFIRSTDOC / REJECTFIRSTCERT; возврат к рассмотрению возможен только после повторной загрузки документов (для REJECTFIRSTDOC/REJECTFIRSTCERT).
  - При отклонении МВД пользователь переходит в REJECTSECOND и is_active = False, а не возвращается к PENDINGTOFIRST; связь «rejected → DB3» для МВД вводит в заблуждение.
  - Push отправляется также после загрузки документов (documents_uploaded), при одобрении/отклонении финансиста и МВД; на диаграмме FCM_REG показан только от DB5 и DB7.

### 5.3. Потенциальные проблемы

- **Ручные проверки без SLA**: время между PENDINGTOFIRST и решением финансиста/МВД не ограничено; пользователь может долго ждать; целесообразно описать в ТЗ ожидаемые сроки или уведомления «заявка на проверке».
- **Безопасность загрузки документов**: проверка типов файлов, лимиты размера, антивирус/сканирование в коде могут быть не показаны; в ТЗ стоит зафиксировать требования к валидации и хранению в MinIO (доступ, retention).
- **Повторная отправка SMS**: при частых запросах с одного номера rate limit снижает риск спама; возможны краевые случаи при смене IP.
- **Один Application на пользователя**: в коде связь User — Application один к одному (uselist=False); при повторной загрузке документов после отклонения обновляется та же заявка (сброс статусов). Диаграмма это не уточняет.
- **Идемпотентность upload-documents**: повторный вызов с теми же данными перезаписывает документы и при определённых условиях сбрасывает статус заявки; дублирование заявок не создаётся.

## 6. Рекомендации по улучшению диаграммы для ТЗ

1. **Уточнить состояния при отказе**  
   Для финансиста: явно показать переход в REJECTFIRST / REJECTFIRSTDOC / REJECTFIRSTCERT (три варианта) и возможность повторной загрузки документов только для REJECTFIRSTDOC и REJECTFIRSTCERT с возвратом в PENDINGTOFIRST. Для МВД: показывать переход в **REJECTSECOND** (отдельный блок) с деактивацией и блокировкой входа, без возврата к DB3.

2. **Добавить push после загрузки документов**  
   Связь от S3 (или от блока «Загрузка документов») к FCM_REG с подписью «documents_uploaded».

3. **Показать блокировку REJECTSECOND**  
   На шаге send_sms и verify_sms добавить условие: «если по номеру есть REJECTSECOND → 403, отказ в регистрации».

4. **Отдельно выделить поручительство**  
   Ветка: пользователь в статусе REJECTFIRST привлекает гаранта; после принятия заявки гарантом заявка может быть передана на проверку МВД; при одобрении МВД роль остаётся REJECTFIRST, но доступ к аренде разрешён при наличии активного гаранта.

5. **Добавить «запрос на повторную проверку»**  
   Действие финансиста: возврат заявки в PENDING (financier_status, mvd_status), роль пользователя → PENDINGTOFIRST; push пользователю о необходимости повторной проверки документов.

6. **Указать создание Application**  
   Явно отметить, что запись Application создаётся при первой успешной загрузке документов (в том же шаге, что и DB3); при повторной загрузке после REJECTFIRSTDOC/REJECTFIRSTCERT обновляются статусы существующей заявки.

7. **Опционально: email-верификация и rate limiting**  
   При необходимости ТЗ можно дополнить: отправка кода на email при указании email в send_sms; ограничения на частоту SMS по номеру и IP.

Внесение этих правок сделает диаграмму согласованной с реализацией и пригодной для ТЗ и приёмки.
