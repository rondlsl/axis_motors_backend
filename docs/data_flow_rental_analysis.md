# Анализ Data Flow: Поток аренды автомобиля

Документ подготовлен в рамках разработки технического задания на backend проекта **AZV Motors** (каршеринг в Алматы). Разбирается Data Flow диаграмма процесса аренды от выбора автомобиля до завершения поездки и даётся критическая оценка с учётом реальной архитектуры (алгоритмы домена «Аренда и биллинг» в `docs/business_logic_algorithms.md`).

---

## 1. Общее описание процесса

На диаграмме показан **сквозной сценарий аренды по минутам** (или по часам/дням) в упрощённом виде: от просмотра доступных автомобилей до завершения поездки с финальным расчётом. Покрыты следующие бизнес-сценарии:

- **Выбор авто** — получение списка свободных машин (и при необходимости координат с GPS).
- **Бронирование** — создание резерва с проверкой баланса и фиксацией статусов в БД.
- **Старт аренды** — переход в поездку (в т.ч. проверка selfie), разблокировка ТС через GPS.
- **Поездка** — поминутное списание средств фоновой задачей биллинга; при нулевом балансе — push-уведомление (и в реальной системе — последующая блокировка двигателя).
- **Завершение** — блокировка ТС через GPS, финальный расчёт, перевод аренды и машины в конечные статусы.

Диаграмма **не покрывает**: отмену бронирования, продление аренды, доставку машины механиком, предварительные (scheduled) бронирования, пополнение баланса через ForteBank, штрафы за доставку и платное ожидание в деталях. Внешний платёжный провайдер (Forte) указан в блоке External, но в потоке данных задействован только косвенно (баланс кошелька уже пополнен до бронирования).

## 2. Участники (Actors и компоненты)

| Участник | Роль в потоке |
|----------|----------------|
| **Мобильное приложение (Client)** | Инициатор действий: выбор авто, бронирование, старт, завершение. Отображает данные об аренде и балансе, получает push. |
| **Backend API** | Обработка HTTP-запросов, бизнес-логика (проверки, расчёты, смена статусов), вызов внешних сервисов (GPS, при необходимости Forte), запуск фоновой задачи биллинга. |
| **PostgreSQL** | Хранение: автомобили (`cars`), аренды (`rental_history`), транзакции кошелька (`wallet_transactions`), пользователи (баланс). Единственный источник истины по статусам и балансу. |
| **GPS (внешний сервис)** | Получение координат ТС; выполнение команд разблокировки/блокировки двигателя (unlock/lock). В коде — GlonassSoft API (VEHICLES_API_URL, auth, execute_gps_sequence). |
| **ForteBank (внешний)** | Пополнение кошелька пользователя (вебхуки/верификация платежа). В диаграмме показан как участник, но в потоке аренды участвует опосредованно: баланс проверяется при бронировании и списывается при старте и в billing_job. |
| **FCM (push)** | Отправка уведомлений пользователю (низкий баланс, нулевой баланс, блокировка двигателя и т.д.). В проекте реализовано через Expo Push API. |

## 3. Пошаговый разбор потока

### Шаг 1: Выбор авто

| Аспект | Описание |
|--------|----------|
| **Действие клиента** | Пользователь открывает карту/каталог и запрашивает список доступных автомобилей. |
| **Вызов API** | На диаграмме: `GET /vehicles/`. В коде бэкенда список машин может формироваться через внутренний сервис или прокси к внешнему VEHICLES_API_URL (`/vehicles/`); для отображения на карте используются координаты из GPS. |
| **БД** | Чтение: автомобили со статусом `FREE` (и при необходимости другие статусы для отображения). В реализации используются также `Car.status`, фильтрация по классу доступа пользователя. |
| **Внешние сервисы** | **GPS**: актуальные координаты ТС для отображения на карте. |
| **Статусы** | Не меняются; используются только для фильтрации (показать свободные машины). |

**Уточнение по коду**: В `app` список для клиента может собираться через WebSocket (`get_vehicles_data_for_user`) или отдельный эндпоинт, обращающийся к VEHICLES_API_URL; прямой эндпоинт `GET /vehicles/` в роутере rent не описан — уточнить в ТЗ, какой именно URL использует мобильное приложение.

---

### Шаг 2: Бронирование

| Аспект | Описание |
|--------|----------|
| **Действие клиента** | Выбор машины, типа аренды (минуты/часы/дни), при необходимости — доставки; нажатие «Забронировать». |
| **Вызов API** | На диаграмме: `POST /rent/booking/advanced`. В коде: **`POST /rent/reserve-car/{car_id}`** (мгновенное бронирование) или **`POST /rent/advance-booking`** (предварительное с `scheduled_start_time`). Эндпоинта `booking/advanced` в точности нет — диаграмму лучше привести к фактическим путям. |
| **БД** | **Чтение**: Car (статус FREE, тарифы), User (wallet_balance, класс доступа), при необходимости — конфликтующие брони. **Запись**: создание `RentalHistory` (rental_status=**RESERVED** или DELIVERY_RESERVED/DELIVERING при доставке); обновление `Car`: status=**RESERVED**, current_renter_id=user_id. При наличии rebooking_fee — списание с баланса и запись `WalletTransaction`. |
| **Проверка баланса** | Требуемый баланс считается через `calc_required_balance` (open_fee, минуты/часы/дни, доставка, топливо). При `wallet_balance < total_required_balance` возвращается 402 (InsufficientBalanceException). |
| **Внешние сервисы** | ForteBank в этом шаге не вызывается — проверяется уже имеющийся баланс кошелька (который мог быть пополнен ранее через Forte). |

**Условия и ветвления**: нет активной аренды у пользователя (RESERVED/IN_USE/DELIVERING); машина FREE; класс авто доступен пользователю (`check_user_trips_for_class_access`); при advance-booking — проверка конфликтов по времени.

---

### Шаг 3: Старт аренды

| Аспект | Описание |
|--------|----------|
| **Действие клиента** | Подтверждение старта поездки; при необходимости — загрузка selfie и фото авто (photos_before). |
| **Вызов API** | На диаграмме: `POST /rent/start-rental`. В коде: **`POST /rent/start/{car_id}`**. |
| **Selfie-проверка** | На диаграмме указано «selfie проверка» с циклом к A3: в реализации это не повторный вызов того же эндпоинта, а **верификация selfie внутри одного запроса** (сравнение с фото в профиле). При провале верификации запрос возвращает ошибку; при успехе — продолжается та же транзакция. Цикл на диаграмме можно трактовать как «повторная попытка пользователя после загрузки нового selfie». |
| **БД** | **Чтение**: активная аренда (RESERVED) по user_id и car_id; Car; User (баланс). **Запись**: списания (RENT_BASE_CHARGE, open_fee, при доставке delivery_fee) через `record_wallet_transaction`; обновление `User.wallet_balance`; `RentalHistory`: rental_status=**IN_USE**, start_time, already_payed, URL фото; `Car`: status=**IN_USE**, current_renter_id=user_id. |
| **Внешние сервисы** | **GPS**: команда разблокировки двигателя (`execute_gps_sequence(imei, auth_token, "start")` или `unlock_engine`). |

**Условия**: аренда в статусе RESERVED и принадлежит пользователю; баланс достаточен для старта (база + open_fee + delivery_fee для часового/суточного); для владельца — минимум списаний.

---

### Шаг 4: Поездка и биллинг

| Аспект | Описание |
|--------|----------|
| **Действие клиента** | Пользователь управляет автомобилем; приложение может показывать баланс и предупреждения. Клиент не вызывает эндпоинт «поездка» — биллинг идёт в фоне. |
| **Вызов API** | На диаграмме: **`billing_job (каждые 60 сек)`**. В коде: задача по расписанию (APScheduler, интервал 60 сек), внутри — `billing_job` → `_billing_job_impl` → `process_rentals_sync` (в потоке). Используется распределённый лок `BillingLock`, чтобы только один инстанс выполнял биллинг. |
| **БД** | **Чтение**: аренды в статусах RESERVED, IN_USE, DELIVERING, DELIVERY_RESERVED, DELIVERING_IN_PROGRESS; User, Car, тарифы. **Запись**: для **IN_USE** по минутам — списание за каждую минуту (не более 1 минуты за один запуск job), тип транзакции RENT_MINUTE_CHARGE; обновление `rental_history` (base_price/overtime_fee, total_price, already_payed), `User.wallet_balance`; новые записи `wallet_transactions`. Для RESERVED — платное ожидание после 15 мин (RENT_WAITING_FEE). |
| **Условие «баланс < 0»** | В `process_rentals_sync`: при нулевом или отрицательном балансе у аренды IN_USE фиксируется время; через 10 минут без пополнения в список `lock_requests` добавляется запрос на блокировку двигателя. Push-уведомления: low_balance (баланс ≤1000₸), balance_exhausted (0₸), при блокировке — engine_locked_due_to_balance. |
| **Внешние сервисы** | **FCM (push)**: отправка накопленных уведомлений после `process_rentals_sync` (батчами, с семафором). При наличии lock_requests вызывается GPS API (send_lock_engine). |

**Потоки данных**: минуты аренды считаются от `rental.start_time`; цена за минуту берётся из тарифа машины; сумма списания = price_per_minute × количество минут (с ограничением 1 минута за запуск job).

---

### Шаг 5: Завершение аренды

| Аспект | Описание |
|--------|----------|
| **Действие клиента** | Нажатие «Завершить поездку»; при необходимости — загрузка фото после поездки, отзыв. |
| **Вызов API** | На диаграмме: `POST /rent/complete-rental`. В коде: **`POST /rent/complete`** (без суффикса -rental). |
| **БД** | **Чтение**: активная аренда IN_USE с блокировкой for_update; Car; транзакции кошелька. **Запись**: перерасчёт овертайма и поминутных сумм; `verify_and_fix_rental_balance` (balance_before/balance_after по транзакциям, приведение user.wallet_balance к расчётному); RentalHistory: rental_status=**COMPLETED**, end_time, total_price, already_payed, фото; Car: status=**FREE** (в коде при завершении выставляется PENDING, затем при передаче механикам/владельцу — FREE; на диаграмме указано FREE как итог); создание/обновление отзыва при наличии. |
| **Внешние сервисы** | **GPS**: команда блокировки двигателя (`execute_gps_sequence(..., "complete_exterior")` / lock). **Финальный расчёт**: не отдельный вызов к ForteBank — итоговая сумма формируется из уже проведённых транзакций (минуты, овертайм, топливо, ожидание) и при необходимости корректируется в `complete`; пополнение кошелька пользователем делается отдельно (add_money / вебхуки Forte). |

**Условия**: перед завершением выполняется проверка состояния ТС (`check_vehicle_status_for_completion`: двигатель, двери, окна, ручник, фары); при ошибках или списке замечаний возвращается 400.

## 4. Потоки данных

| Шаг | Откуда → Куда | Данные / триггеры |
|-----|----------------|-------------------|
| 1 | Client → API | GET-запрос (опционально query: координаты, фильтры). |
| 1 | API → DB | Запрос машин по status=FREE (и др.). |
| 1 | API → GPS | Запрос координат/состояния ТС (если список машин обогащается данными с GPS). |
| 1 | API → Client | JSON: список автомобилей (id, координаты, статус, тарифы, класс и т.д.). |
| 2 | Client → API | POST body: car_id, rental_type, duration, with_driver, delivery (если есть). |
| 2 | API → DB | Проверка баланса (SELECT user.wallet_balance), вставка RentalHistory, UPDATE cars. При rebooking_fee — INSERT wallet_transactions, UPDATE user.wallet_balance. |
| 2 | API → Client | 201 + данные аренды (rental_id, reservation_time, car, …) или 402 при недостатке баланса. |
| 3 | Client → API | POST /rent/start/{car_id}; multipart: selfie, photos_before (опционально). |
| 3 | API → DB | SELECT rental (RESERVED), расчёт сумм, INSERT wallet_transactions, UPDATE rental (IN_USE, start_time), UPDATE car (IN_USE), UPDATE user.wallet_balance. |
| 3 | API → GPS | Команда unlock (imei, auth_token, action). |
| 3 | API → Client | 200 + message, rental_id. |
| 4 | Scheduler → Billing | Триггер каждые 60 сек; захват BillingLock (Redis). |
| 4 | Billing → DB | SELECT активных аренд; для каждой — расчёт минут/овертайма/ожидания, INSERT wallet_transactions, UPDATE rental и user.wallet_balance. |
| 4 | Billing → FCM | Список (user_id, translation_key или title/body); отправка push (low_balance, balance_exhausted, engine_locked и т.д.). |
| 4 | Billing → GPS | При lock_requests — send_lock_engine(imei, auth_token) для каждого зависшего с нулевым балансом 10+ мин. |
| 5 | Client → API | POST /rent/complete; body: отзыв, фото (опционально). |
| 5 | API → GPS | Команда lock (complete_exterior / lock_engine). |
| 5 | API → DB | SELECT rental for_update; проверка ТС; перерасчёт сумм; verify_and_fix_rental_balance; UPDATE rental (COMPLETED, end_time), car (FREE/PENDING), user. |
| 5 | API → Client | 200 + итог аренды, баланс. |

**Идентификаторы**: rental_id (UUID/SID), car_id, user_id; в wallet_transactions — rental_id, amount, type, balance_before/after. **Форматы**: JSON для API; в БД — статусы enum (RentalStatus, CarStatus), суммы в целых (тенге).

## 5. Критическая оценка

### 5.1. Полнота

- **Отражено достаточно**: выбор авто, бронирование с проверкой баланса, старт с selfie и unlock, поминутный биллинг, завершение с lock и финальным расчётом.
- **Не отражено или упрощено**:
  - **Отмена бронирования** (`POST /rent/cancel`): смена RESERVED → COMPLETED, платное ожидание, комиссия, 50% владельцу; Car → FREE.
  - **Продление аренды** (`POST /rent/extend`): доп. списание, обновление duration/planned_end.
  - **Доставка механиком**: статусы DELIVERY_RESERVED, DELIVERING, DELIVERING_IN_PROGRESS; штрафы за задержку доставки в billing_job.
  - **Предварительные бронирования**: advance-booking, scheduled_start_time, процесс активации по расписанию (process_scheduled_bookings).
  - **Пополнение баланса**: ForteBank (add_money, вебхуки) — на диаграмме Forte указан, но поток «деньги от банка → кошелёк» не разложен.
  - **Обработка ошибок**: 402 при недостатке баланса, 400 при проверке ТС при завершении, таймауты GPS, повторные попытки.
  - **Платное ожидание**: первые 15 минут бесплатно, далее списание в billing_job (на диаграмме только «проверка баланса» при бронировании и «balance -= price_per_minute» при биллинге).
  - **Car.status**: при завершении в коде выставляется PENDING (машина ожидает осмотра), а не сразу FREE — стоит уточнить на диаграмме.

### 5.2. Соответствие архитектуре

- **Совпадения**: цепочка FREE → RESERVED → IN_USE → COMPLETED; списание в wallet_transactions и обновление balance; billing каждые 60 сек; unlock/lock через GPS; push при низком/нулевом балансе.
- **Расхождения с кодом**:
  - Пути API: в коде нет `POST /rent/booking/advanced` и `POST /rent/start-rental`, `POST /rent/complete-rental` — есть `POST /rent/reserve-car/{car_id}`, `POST /rent/advance-booking`, `POST /rent/start/{car_id}`, `POST /rent/complete`. Рекомендуется привести подпись эндпоинтов на диаграмме в соответствие с реализацией.
  - Список машин: `GET /vehicles/` может относиться к внешнему VEHICLES_API_URL; в backend приложении список для клиента собирается через WebSocket или прокси — нужно зафиксировать в ТЗ.
  - Selfie: на диаграмме цикл «selfie проверка → A3»; в коде это один запрос start с верификацией внутри, без повторного вызова эндпоинта бронирования.

### 5.3. Потенциальные проблемы

- **Race conditions**: два запроса на бронирование одной машины одновременно — в коде проверка Car.status и создание RentalHistory выполняются в одной транзакции; желательно явно указать блокировку (SELECT FOR UPDATE по car_id) или уникальные ограничения.
- **Идемпотентность**: повторная отправка `POST /rent/complete` может привести к повторному переводу в COMPLETED и повторным вызовам GPS lock; в коде есть проверка активной аренды IN_USE — повторный вызов вернёт 404 после первого успешного завершения. Идемпотентность по idempotency-key не описана — для платежных сценариев стоит рассмотреть.
- **Узкие места**: один billing_job на кластер (лок); при большом числе активных аренд один проход process_rentals_sync может занимать заметное время; задержка до 60 сек перед первым поминутным списанием.
- **Двойное списание**: ограничение «не более 1 минуты за запуск» в process_rentals_sync снижает риск при сбоях, но при падении инстанса и переключении лока другой инстанс может обработать те же аренды с небольшой задержкой — дублирование минут маловероятно, но границы окна стоит описать.
- **ForteBank**: на диаграмме не показано, когда и как баланс пополняется (вебхук после оплаты, вызов add_money); при недостатке баланса пользователь должен иметь возможность пополнить счёт — поток «пополнение → повторное бронирование» отсутствует.

## 6. Рекомендации по улучшению и дополнению диаграммы

1. **Привести названия эндпоинтов к коду**  
   Заменить `POST /rent/booking/advanced` на `POST /rent/reserve-car/{car_id}` (и при необходимости отдельно показать `POST /rent/advance-booking`). Заменить `POST /rent/start-rental` на `POST /rent/start/{car_id}`, `POST /rent/complete-rental` на `POST /rent/complete`. Для списка машин указать фактический эндпоинт (например, прокси `GET /vehicles/` или «WebSocket / initial snapshot»).

2. **Добавить альтернативные и ошибочные сценарии**  
   - Ветка «Недостаточно баланса» при бронировании (402) → переход к пополнению (Forte) → повторная попытка бронирования.  
   - Ветка «Отмена бронирования»: RESERVED → POST /rent/cancel → COMPLETED, Car FREE, списание платного ожидания при необходимости.  
   - Ветка «Ошибка проверки ТС при завершении» (400) → пользователь устраняет замечания (закрыть дверь и т.д.) → повторный complete.

3. **Уточнить блок ForteBank**  
   Показать поток: «Клиент инициирует оплату → Forte → вебхук/verify → add_money → wallet_transactions (+), user.wallet_balance» и связь с проверкой баланса при бронировании (достаточность средств после пополнения).

4. **Разделить billing_job на подшаги**  
   Например: «1) Захват BillingLock. 2) process_rentals_sync: списания по минутам, формирование списков push и lock_requests. 3) Отправка push. 4) При balance ≤ 0 и 10+ мин — вызов GPS lock. 5) WebSocket notify.» Это прояснит зависимость «баланс < 0 → FCM» и «баланс < 0 и время → GPS lock».

5. **Selfie**  
   Заменить цикл «selfie проверка → A3» на один блок «Старт аренды (внутри: верификация selfie, списания, unlock)» с выходом «ошибка верификации → 4xx клиенту», чтобы не создавать впечатление повторного вызова бронирования.

6. **Статус Car при завершении**  
   Уточнить на диаграмме: после complete выставляется PENDING (ожидание осмотра), затем при смене жизненного цикла — FREE (или оставить один итоговый статус FREE, если в ТЗ считается иначе).

7. **Обработка ошибок и таймауты**  
   Добавить блоки «Таймаут GPS» и «Повторные попытки unlock/lock» (если есть в коде), а также «Откат транзакции при ошибке в complete» (частичный коммит не допускается за счёт одной транзакции).

8. **Поток данных для биллинга**  
   Явно указать входные данные billing_job: «get_local_time(), активные аренды (статусы …), тарифы из Car» и выходные: «обновлённые rental_history, wallet_transactions, user.wallet_balance; списки push и lock_requests».

Внесение этих правок сделает диаграмму согласованной с реализацией и пригодной для ТЗ и обсуждения с командой.
