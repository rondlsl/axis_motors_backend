# Анализ Data Flow: Поток данных GPS-телеметрии

Документ подготовлен в рамках анализа архитектуры backend проекта **AZV Motors** (каршеринг в Алматы). Разбирается Data Flow диаграмма сбора и распространения телеметрии автомобилей в реальном времени и даётся критическая оценка с учётом реализации в коде (main.py, telemetry_cache, WebSocket, push).

---

## 1. Общая схема сбора телеметрии

### 1.1. Источники данных

| Уровень | Описание |
|--------|-----------|
| **GPS-трекеры в авто (Трекер 1 … N)** | Устройства в автомобилях передают координаты, уровень топлива, пробег и др. во внешнюю облачную платформу. В коде AZV Backend трекеры не вызываются напрямую — данные приходят через промежуточный API. |
| **GlonassSoft Cloud** | Облачный сервис GlonassSoft принимает данные с трекеров и предоставляет API для мониторинга (например, devicestatebyimei для детальной телеметрии по IMEI). Используется для команд (unlock/lock engine) и для эндпоинта GET /vehicles/telemetry/{car_id} (админ). |
| **Vehicles API (195.93.152.69:8667)** | Внешний HTTP-сервис, доступный по `VEHICLES_API_URL` (в конфиге — этот хост:порт или аналог). Отдаёт агрегированный список машин с последними данными телеметрии. |

Поток: **Трекеры → GlonassSoft Cloud → Vehicles API**. AZV Backend опрашивает только **Vehicles API**; прямой вызов GlonassSoft в периодической задаче обновления списка машин не используется — только для точечных запросов (телеметрия по car_id, отправка команд).

### 1.2. Роль Vehicles API как кэширующего прокси

На диаграмме блок «Кеш GPS данных» отнесён к Vehicles API. Фактически:

- **Vehicles API** выступает единой точкой входа для получения «снимка» по всем машинам: один HTTP GET возвращает список с полями `vehicle_id`, `latitude`, `longitude`, `fuel_level`, `mileage` (и при необходимости `vehicle_imei`, `speed`). Тем самым Backend не обращается к GlonassSoft по каждой машине каждую секунду — нагрузка на GlonassSoft снижается.
- **Кэш на стороне AZV Backend** реализован отдельно — в **Redis** (`app/services/telemetry_cache.py`). В нём хранятся последние известные значения (lat, lon, fuel, mileage, last_db_update) по каждому `gps_id`. Он используется не для кэширования ответа Vehicles API, а для **решения, нужно ли писать в PostgreSQL**: обновление БД выполняется только при «значимом» изменении (пороги по расстоянию, топливу, времени). Таким образом, «Кеш GPS данных» на диаграмме можно трактовать как кэш/агрегат на стороне Vehicles API; второй уровень оптимизации — наш Redis + пороги значимых изменений.

## 2. Периодическая задача (Scheduled Job)

### 2.1. Триггер

- **Интервал**: каждые **1 секунду**.
- **Регистрация**: `scheduler.add_job(check_vehicle_conditions, "interval", seconds=1, coalesce=True)` в `main.py` (startup).
- **Функция**: `check_vehicle_conditions()` просто вызывает `await update_vehicle_data()`; вся логика в `update_vehicle_data()`.

### 2.2. Действия

1. **Получение данных**: `vehicles_data = await get_last_vehicles_data()`.
   - Внутри: HTTP GET к `{VEHICLES_API_URL}/vehicles/?skip=0&limit=100`, timeout 5 с, `response.json()`.
   - Возвращается список словарей; при любой ошибке или не-2xx возвращается пустой список `[]`, цикл обновления не выполняется.
2. **Подготовка кэша**: из ответа собираются `gps_id` (поле `vehicle_id`); вызывается `load_telemetry_batch(gps_ids)` — загрузка из Redis последних значений по каждому gps_id.
3. **Синхронная обработка в потоке**: `await loop.run_in_executor(None, _update_in_thread, vehicles_data, telemetry_cache, loop)`.
   - `_update_in_thread` создаёт сессию БД (`get_db`), вызывает `_update_vehicle_data_sync(vehicles_data, db, telemetry_cache, loop)` и в `finally` закрывает сессию.
4. **Парсинг ответа**: в `_update_vehicle_data_sync` для каждого элемента списка читаются ключи `vehicle_id`, `latitude`, `longitude`, `fuel_level`, `mileage`. Соответствие Car в БД — по `Car.gps_id == vehicle_id` (строка).

### 2.3. Использование run_in_executor

- **Причина**: работа с PostgreSQL (много последовательных запросов и commit) выполняется в синхронном коде; чтобы не блокировать event loop, она выносится в поток пула по умолчанию через `run_in_executor`.
- **Передача loop**: в поток передаётся `loop` для вызова асинхронной отправки push при детекции заправки: `asyncio.run_coroutine_threadsafe(_send_fuel_notification(), loop)`.

## 3. Обновление данных в PostgreSQL

### 3.1. Какие поля таблицы `cars` обновляются

Обновляются только при **значимом изменении** (решение по `is_significant_change` + запись в кэш при каждом проходе):

| Поле | Условие обновления |
|------|--------------------|
| **latitude** | Новые значения не None и не (0, 0); при значимом изменении записываются. |
| **longitude** | Аналогично. |
| **updated_at** | Выставляется `get_local_time()` при обновлении координат. |
| **fuel_level** | При значимом изменении; перед записью выполняется проверка на заправку (см. ниже). |
| **mileage** | При значимом изменении, если `mileage is not None`. |

**Скорость (speed)** в этой задаче в таблицу `cars` **не пишется**. Скорости при необходимости берутся при формировании ответа WebSocket: отдельный запрос к `VEHICLES_API_URL/vehicles` в `get_vehicles_data_for_user`, маппинг по `vehicle_imei`/gps_imei.

### 3.2. Массовое или построчное обновление

- **Построчное**: в цикле по каждому элементу `vehicles_data` находится одна запись `Car` по `gps_id`; при `should_update == True` изменяются атрибуты этого объекта (latitude, longitude, fuel_level, mileage, updated_at). Один общий `db.commit()` после цикла — то есть одна транзакция на весь проход, но логически обновления по одной машине за итерацию.
- На диаграмме формулировка «UPDATE cars SET lat, lon, fuel, mileage» по смыслу верна (обновляются эти поля), но в коде используется ORM (присвоение полям объекта Car), а не сырой SQL с массовым UPDATE.

## 4. Детекция событий

### 4.1. Заправка: условие «fuel +10%»

- **Условие**: не «увеличение на 10% от текущего значения» как единственный критерий, а комбинация:
  - Уже принято решение о значимом изменении и обновлении БД (`should_update == True`).
  - В блоке обновления `fuel_level`: если `old_fuel` (текущее в Car) не None и > 0, и новый `fuel > old_fuel`, то считаются прирост и процент:  
    `fuel_increase_percent = (fuel - old_fuel) / old_fuel * 100`.  
  - **Если fuel_increase_percent > 10** — считается заправка.
- **Действие**: поиск активной аренды по `car.id` (`get_active_rental_by_car_id`); если аренда есть — берётся пользователь (арендатор); в основной поток через `run_coroutine_threadsafe` ставится корутина отправки push с ключом `fuel_refill_detected` (локализованное уведомление «обнаружена заправка»). Отправка выполняется с учётом глобального семафора push.
- На диаграмме «fuel +10%?» и «FCM: Уведомление о заправке» соответствуют этой логике; уточнение: порог именно **> 10%** от предыдущего уровня топлива.

### 4.2. Другие события

В данной задаче явно не детектируются: превышение скорости, выход из зоны, зажигание и т.д. Они могут обрабатываться в других местах (например, при проверке завершения аренды или в GlonassSoft).

## 5. WebSocket-трансляция

### 5.1. Вызов notify_vehicles_list_update

- **Когда**: только если за итерацию был хотя бы один запись в БД (`updated > 0`). Вызов: `asyncio.create_task(notify_vehicles_list_update())` — без аргументов, то есть рассылка **всем** подключённым по каналу списка машин.
- **Реализация**: в `notify_vehicles_list_update(user_id=None)` при `user_id is None` получаются все подключённые к `connection_type="vehicles_list"` через `get_connected_users("vehicles_list")`; для каждого пользователя создаётся сессия БД, загружается User, вызывается `get_vehicles_data_for_user(user, db)` и отправляется персональное сообщение с этими данными.

### 5.2. Какие данные передаются и формат сообщения

- **Тип сообщения**: `type: "vehicles_list"`.
- **Тело**: `{ "type": "vehicles_list", "data": vehicles_data, "timestamp": "<isoformat>" }`.
- **vehicles_data**: результат `get_vehicles_data_for_user(user, db)` — список машин, видимых данному пользователю (с учётом роли, активной аренды, класса авто). Для каждой машины формируются поля: id (sid), координаты (latitude, longitude), статус, топливо, пробег, имя, госномер, фото, цена открытия, скорость (если получена из Vehicles API при формировании ответа) и др. Скорости при формировании списка запрашиваются отдельным GET к `VEHICLES_API_URL/vehicles` и маппятся по `gps_imei`.

Таким образом, клиенты (Mobile App, Admin Panel) получают не «сырой» срез телеметрии, а уже отфильтрованный и обогащённый список машин под свою роль и контекст.

## 6. Критическая оценка

### 6.1. Нагрузка

- **Опрос каждые 1 сек**: один GET на `VEHICLES_API_URL/vehicles/?skip=0&limit=100` в секунду. Для одного инстанса Backend это обычно допустимо; при нескольких инстансах каждый будет опрашивать независимо — дублирование запросов к Vehicles API.
- **Rate limiting**: в коде AZV Backend явного ограничения частоты запросов к Vehicles API нет; таймаут 5 с. Ограничения, если есть, задаются на стороне Vehicles API или GlonassSoft.
- **Снижение нагрузки на БД**: за счёт Redis-кэша и порогов значимых изменений число UPDATE в БД меньше числа опросов (логируется статистика db_updates vs total_updates и % reduction). Пороги: координаты > 1 м (в коде комментарий про 10 м в docstring — фактически в `is_significant_change` используется 1 м), топливо > 0.5 л, принудительный sync раз в 60 сек.

### 6.2. Отказоустойчивость

- **Недоступность Vehicles API**: при исключении в `get_last_vehicles_data()` возвращается пустой список; `update_vehicle_data()` выходит без обновлений и без вызова WebSocket. Ошибка логируется (`logger.error`). Повторная попытка — через 1 сек по расписанию. Явного backoff или счётчика ошибок под увеличение интервала нет.
- **Недоступность Redis**: в `telemetry_cache` при недоступности Redis `load_telemetry_batch` возвращает пустой кэш; тогда для каждой машины `cached` — пустой TelemetryData, и `is_significant_change` часто возвращает True (first_update / no_previous_db_update), то есть **все данные пишутся в БД**. То есть деградация «без Redis» ведёт к росту числа записей в БД, но функционально поток продолжает работать.
- **Ошибка в _update_vehicle_data_sync**: при исключении выполняется `db.rollback()`, возвращается (0, []); WebSocket не вызывается; кэш для этой итерации не обновляется.

### 6.3. Актуальность данных

- **1 сек** для каршеринга обычно достаточно для отображения машин на карте и базового мониторинга (топливо, пробег). Более частый опрос упирается в нагрузку на Vehicles API и в то, что трекеры сами могут отдавать данные с меньшей частотой.
- Задержка до клиента: 1 сек (опрос) + время обработки + время доставки WebSocket. При `updated > 0` клиенты получают обновление в течение того же цикла (create_task без ожидания).

### 6.4. Соответствие архитектуре

- Диаграмма в целом совпадает с описанным в `docs/business_logic_algorithms.md` модулем GPS/телеметрии (алгоритмы G1, G2, G3, G4, WebSocket vehicles_list): опрос каждую секунду, get_last_vehicles_data → Vehicles API, run_in_executor, обновление Car по значимым изменениям, детекция заправки >10%, notify_vehicles_list_update.
- **Уточнения**: (1) между GlonassSoft и Vehicles API на диаграмме связь «GS1 → VA» — в коде Backend напрямую к GlonassSoft в этой задаче не обращается; (2) кэш на диаграмме показан в блоке Vehicles API, тогда как в коде ещё есть Redis-кэш на стороне Backend; (3) WebSocket вызывается только при updated > 0, а не при каждом запуске задачи.

## 7. Рекомендации

1. **Уточнить диаграмму**  
   - Явно показать **Redis** (Backend) как второй уровень кэша и место, где решается вопрос «писать ли в БД».  
   - Указать, что **notify_vehicles_list_update** вызывается только при `updated > 0`.  
   - При желании разделить «Vehicles API» (внешний сервис) и «Кеш GPS данных» (их внутренний кэш), если это разные компоненты у заказчика.

2. **Обработка ошибок Vehicles API**  
   - Рассмотреть экспоненциальный backoff или увеличение интервала при повторяющихся ошибках (например, после 3–5 подряд пустых/ошибочных ответов увеличить интервал до 5–10 сек с последующим возвратом к 1 сек при успехе).  
   - Опционально: алерт в Telegram при длительной недоступности Vehicles API.

3. **Оптимизация при нескольких инстансах**  
   - Если Backend работает в несколько реплик, каждая будет запускать свою задачу раз в 1 сек и отдельно опрашивать Vehicles API. Можно ввести распределённый лок (аналогично billing_job), чтобы опрос и обновление выполнял только один инстанс; остальные могут не запускать задачу или только подписываться на события (если появится общий канал обновлений).

4. **Обновление только изменившихся данных**  
   - Уже реализовано за счёт `is_significant_change` и записи в БД только при значимых изменениях. При необходимости можно дополнительно сузить круг обновляемых полей (например, не трогать updated_at при изменении только пробега, если это не нужно для бизнес-логики).

5. **Сохранение сырой телеметрии**  
   - Сейчас в БД хранятся только последние значения в `cars`. Для аналитики или аудита можно рассмотреть запись снимков телеметрии в отдельную таблицу или во временное хранилище с TTL (например, раз в минуту или при значимом изменении).

6. **Детекция заправки**  
   - Порог 10% за одну итерацию (1 сек) корректен для фильтрации шума; при появлении требований к другим сценариям (например, «дозаправка менее 5%») можно вынести порог в конфиг.

Внесение этих уточнений и рекомендаций улучшит согласованность диаграммы с реализацией и пригодность архитектуры к масштабированию и эксплуатации.
